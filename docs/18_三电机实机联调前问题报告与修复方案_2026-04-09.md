# 三电机实机联调前问题报告与修复方案（2026-04-09）

当前系统准备以 `Eyou_ROS1_Master` 作为统一外观层，先开展 3 电机实机联调：

- `shoulder_yaw_joint`：来自 `Eyou_Canopen_Master`
- `shoulder_pitch_joint`：来自 `Eyou_Canopen_Master`
- `elbow_pitch_joint`：来自 `can_driver`

正式模型统一采用：

- `car_urdf/urdf/car_urdf.urdf`

并以 URDF 中的关节命名作为系统唯一标准名。

本报告用于固定：

1. 当前链路里哪些问题会阻碍 3 电机实机联调
2. 哪些问题只是“后续扩展问题”，当前可暂不处理
3. 后续应按什么顺序修

## 3. 当前仍存在的 3 个关键问题

### 3.1 问题 1：`can_driver` 当前仍指向 `can0`

当前配置中：

- `Eyou_Canopen_Master/config/joints.yaml`
  - `canopen.interface = can0`
- `can_driver/config/can_driver.yaml`
  - `elbow_pitch_joint.can_device = can0`

这意味着：

- 现在两边实际上仍打到同一条总线

若当前实机联调目标是：

- `Eyou_Canopen_Master -> can0`
- `can_driver -> can1`

那么这一项必须先调整，否则 `can_driver` 侧会初始化到错误总线。

#### 影响

- `hybrid ~/init` 会打错物理 CAN 口
- 现场抓包和故障定位会被严重干扰
- 可能误以为是生命周期或驱动问题，实际只是总线配置错位

#### 修复

将：

- `can_driver/config/can_driver.yaml`
  中 `elbow_pitch_joint.can_device`

从：

- `can0`

改为：

- `can1`

#### 优先级

最高。  
这项不改，不建议开始实机上电联调。

---

### 3.2 问题 2：`hybrid` 上的 Ruckig executor 目前只覆盖 CANopen 关节

当前 `Eyou_ROS1_Master` 虽然已经集成：

- `IpFollowJointTrajectoryExecutor`

但在 `hybrid_motor_hw_node.cpp` 中，executor 的 `exec_cfg` 仍然只从：

- `Eyou_Canopen_Master/config/joints.yaml`

构造，也就是只包含：

- `shoulder_yaw_joint`
- `shoulder_pitch_joint`

没有把 `can_driver` 侧的：

- `elbow_pitch_joint`

纳入统一 action 执行器。

#### 影响

当前即使打开：

- `use_ip_executor=true`

也只能得到：

- 两轴 CANopen executor

而不是：

- 三电机 unified executor

这会导致：

1. `hybrid` action 入口和当前实际联调目标不一致
2. UI 或 MoveIt 若发三轴 goal，会在 joint 集不匹配处失败
3. 后续一旦扩关节数量，又需要继续补代码

#### 正确修复方向

Ruckig executor 必须保持**动态配置**，不能把“三电机列表”写死在 `master` 包代码里。

应改为：

1. `Eyou_Canopen_Master` 继续在自己的 `joints.yaml` 中维护每轴 Ruckig 约束：
   - `ip_max_velocity`
   - `ip_max_acceleration`
   - `ip_max_jerk`
   - `ip_goal_tolerance`
2. `can_driver` 也在自己的 `can_driver.yaml` 中为每个可做轨迹的 joint 增加同样 4 个字段
3. `Eyou_ROS1_Master` 在运行时动态读取两边配置，合并生成统一的：
   - `IpFollowJointTrajectoryExecutor::Config`
4. executor 的关节集合由“当前实际启用的轨迹关节”自动决定，而不是硬编码

#### 优先级

高。  
如果本轮只做 `~/init/~enable/~resume` 联调，可暂缓；  
如果要做三电机动作联调或 UI 联调，这项必须先修。

---

### 3.3 问题 3：`hybrid` 默认只暴露了 joint_state_controller，没有把 Ruckig action 入口整理成正式联调用法

当前 `hybrid` 默认 launch：

- 只保证 `joint_state_controller` 正常启动

而：

- `arm_position_controller/follow_joint_trajectory`

这条动作入口在 `hybrid` 侧还没有整理成“当前正式联调入口说明”。

同时，`Eyou_Canopen_Master` 现有 UI：

- `scripts/joint_action_ui.py`

虽然本身支持：

- `--joints-yaml`
- `--service-ns`
- `--action-ns`

但它当前默认语义仍面向：

- `canopen_hw_node`

而不是：

- `hybrid_motor_hw_node`

#### 影响

1. 现场虽然已有 UI 资产，但不能直接无脑用于 `hybrid`
2. 使用者容易误连到 `canopen_hw_node` 原生 service/action
3. 造成“胶水包联调”仍然走回 backend 原生入口的旁路

#### 正确修复方向

不需要重写新的大 UI。

应采用：

- wrapper 脚本方案

具体做法：

1. 保留 `Eyou_Canopen_Master/scripts/joint_action_ui.py`
2. 在 `Eyou_ROS1_Master` 中新增一个 wrapper 脚本，例如：
   - `scripts/hybrid_joint_action_ui.py`
3. wrapper 在启动时：
   - 读取 `can_driver/config/can_driver.yaml`
   - 读取 `Eyou_Canopen_Master/config/joints.yaml`
   - 动态生成一个 merged joints yaml
   - 调用原 `joint_action_ui.py`
4. wrapper 默认参数改为：
   - `--service-ns /hybrid_motor_hw_node`
   - `--action-ns /arm_position_controller/follow_joint_trajectory`

这样：

- UI 资产复用
- 入口 authority 仍保持在 `hybrid`
- joint 列表也能随配置动态变化

#### 优先级

高。  
如果本轮需要“边看状态边发三轴动作”，这项应与问题 2 一起修。

## 4. 当前不属于本轮阻塞的问题

以下问题存在，但不阻塞“3 电机实机联调”首轮：

1. `car_urdf` 定义的是 7 个机械臂/夹爪关节，当前运行时只接了其中 3 个
2. MoveIt / 全 6+1 轴 FollowJointTrajectory 全链路尚未打通
3. `can_driver` 未来若扩成多 CAN 设备并行（如 `can1/can3/can5`），`hybrid` init 仍需进一步扩展多 device prepare/init 语义

这些都属于下一阶段问题，不是本轮三电机 bringup 的前置阻塞项。

## 5. 推荐修复顺序

建议严格按下面顺序推进：

### Commit A

`fix(can_driver): move elbow joint to can1`

目标：

- 先保证两边总线配置正确

### Commit B

`feat(can_driver): add per-joint IP executor limits`

目标：

- 让 `can_driver` 侧也具备和 `canopen` 一致的 Ruckig 约束来源

### Commit C

`feat(Eyou_ROS1_Master): build hybrid IP executor config from both backends`

目标：

- 让 `hybrid` executor 真正覆盖当前全部轨迹关节

### Commit D

`fix(Eyou_ROS1_Master): expose hybrid IP executor action path in launch`

目标：

- 把 `use_ip_executor=true` 的正式联调行为整理清楚
- 像 `Eyou_Canopen_Master` 一样明确 action 暴露口径

### Commit E

`feat(Eyou_ROS1_Master): add wrapper for reusing joint_action_ui`

目标：

- 复用现有 UI
- 不绕开 `hybrid`
- 支持动态 joint 集

## 6. 实机联调建议口径

在修完问题 1 之后，可先做最小实机验证：

1. 启动 `Eyou_ROS1_Master/hybrid_motor_hw.launch`
2. 确认：
   - `joint_state_controller` 为 `running`
   - `/joint_states` 有 3 个关节
3. 调：
   - `~/init`
   - `~/enable`
   - `~/resume`
4. 观察：
   - 返回消息
   - `/joint_states`
   - CAN 抓包

若要做“动作联调/UI 联调”，则应在问题 2、3 修完后再进入下一阶段。

## 7. 结论

当前 `Eyou_ROS1_Master` 的胶水包链路已经具备：

- 正式 URDF
- 正式 joint name
- 正式 unified service
- 正式 controller_manager 入口

但距离“3 电机统一动作联调”还差 3 个关键修复：

1. `can_driver` 切到 `can1`
2. hybrid Ruckig executor 动态纳入 `can_driver` 关节
3. 以 wrapper 方式复用现有 UI，统一走 `hybrid` action/service 入口

这 3 项补齐后，才算真正具备 3 电机胶水包动作联调条件。
