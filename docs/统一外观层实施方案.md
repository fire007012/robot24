建议正文要点如下：

#### 1. 背景与目标
- 当前系统存在两套电机控制包：`can_driver` 与 `Eyou_Canopen_Master`
- 两者底层实现不同，配置风格不同，但上层希望尽量不感知差异
- 本方案目标不是统一底层协议实现，而是在保留两套后端的前提下提供统一外观层

#### 2. 设计原则
- 不统一原生 YAML
- 不重写底层协议栈
- 一个统一节点对外暴露系统能力
- 一个统一 `controller_manager`
- joint 命名以现有原生配置中的 `name` 为准
- 旧接口尽量兼容，支持平滑迁移

#### 3. 统一外观范围
统一的内容：
- 统一节点
- 统一启动入口
- 统一控制器配置
- 统一服务入口
- 统一 `/joint_states`
- 统一 diagnostics 入口
- 统一生命周期状态模型与服务语义

不统一的内容：
- `can_driver` 的 MT / PP 协议实现
- `Eyou_Canopen_Master` 的 CANopen / Lely / CiA402 实现
- 两边底层协议状态机与故障恢复细节实现
- 两边原生 YAML 结构

#### 4. 配置与启动方式
- 新增统一 launch，例如 `hybrid_motor_hw.launch`
- launch 只收原生配置文件路径参数：
  - `can_driver_config_path`
  - `canopen_joints_path`
  - `canopen_dcf_path`
  - `controllers_file`
  - 可选 `robot_urdf`
- 统一节点内部分别调用两边现有解析与初始化逻辑
- 不做统一 YAML 到原生 YAML 的转换

#### 5. 节点与硬件组合方案
- 新增统一节点 `hybrid_motor_hw_node`
- 节点内部包含两个 backend adapter：
  - `CanDriverBackendAdapter`
  - `CanopenBackendAdapter`
- 新增组合 `RobotHW`：
  - 作为总的 `hardware_interface::RobotHW`
  - 内部挂接两个子后端
  - 对 `controller_manager` 暴露统一 joint 资源集合
- 启动时校验所有 joint 名全局唯一
- 若任一 backend 初始化失败，统一节点整体启动失败，不做部分降级运行

#### 6. 对外接口方案
统一公开服务：
- `init`
- `shutdown`
- `recover`
- `enable`
- `disable`
- `halt`
- `resume`
- `set_mode`
- `set_zero`

统一生命周期状态：
- `Configured`
- `Standby`
- `Armed`
- `Running`
- `Faulted`
- `Recovering`
- `ShuttingDown`

状态语义约定：
- `init`：建立通信并完成启动检查，成功后进入 `Armed`
- `enable`：从 `Standby` 进入 `Armed`，只表示允许进入可运行待命态，不直接开始运动
- `resume`：从 `Armed` 进入 `Running`，开始接受正常运动命令
- `halt`：从 `Running` 回到 `Armed`，立即停止运动但保持通信与受控待命
- `disable`：从 `Standby` / `Armed` / `Running` 回到 `Standby`，关闭使能或输出
- `recover`：仅用于 `Faulted -> Standby`，不得隐式执行 `enable` 或 `resume`
- `shutdown`：停止通信并回到 `Configured`

`can_driver` 改造约束：
- 不再允许 `recover` 直接等价于 `Enable`
- `recover` 成功后必须要求 fresh command，不能让旧命令立即重新生效
- `Running` 之外的状态不得下发正常运动命令
- 旧接口只保留兼容别名，内部统一转发到新的生命周期服务

协议抽象补齐：
- `can_driver` 需要在 `CanProtocol` 增加独立的 fault reset / recover 原语，例如 `ResetFault()`
- 若某协议无法真正清故障，应明确返回失败，而不是退化为 `Enable`
- `MtCan` 与 `EyouCan` 分别实现各自支持的恢复路径；不支持的协议路径在 service 返回中明确说明

兼容策略：
- 保留旧接口别名并转发到统一服务逻辑
- 保留 `can_driver` 风格 per-joint debug topic 作为调试兼容入口
- 文档中明确这些调试 topic 不是正式控制接口
- `can_driver` 原有 `recover` 语义视为 legacy，不再作为“重新使能”的公开语义

控制器方案：
- 新增一份统一 controllers 配置
- `joint_state_controller` 只保留一份
- 位置控制器和速度控制器只保留统一公开名字
- controller 的 `joints:` 直接使用最终公开 joint 名

#### 7. 分阶段实施
第一阶段：
- 在 `can_driver` 内新增生命周期状态枚举与协调器骨架
- 增加 `enable` / `disable` / `halt` / `resume` 服务入口
- 保留旧接口别名，但新旧接口都走同一套协调逻辑
- 禁止 `recover -> Enable` 的旧行为继续作为正式路径

第二阶段：
- 在 `CanProtocol` 增加 `ResetFault()` 等恢复原语
- 为 `MtCan`、`EyouCan` 分别实现故障恢复路径
- `recover` 改为 `Faulted -> Standby`
- 加入恢复后的 command resync / hold 机制

第三阶段：
- 新增统一 launch
- 新增统一节点骨架
- 两套后端可在同一节点中启动
- `/joint_states` 同时包含两边所有 joint

第四阶段：
- 接入统一 `controller_manager`
- 新增统一 controllers 配置
- 打通位置/速度控制链路

第五阶段：
- 接入统一 service gateway
- 保留旧接口兼容别名
- 增加统一 diagnostics 汇总

第六阶段：
- 完成回归测试
- 旧独立入口改为兼容模式或标记为 legacy

#### 8. 验证与验收
最小验收标准：
- 统一节点能同时加载两边原生配置
- `/joint_states` 中包含全部 joint，名字与原生配置一致
- controller 能同时引用两边的 joint
- 统一 `init/enable/disable/halt/resume/recover/shutdown` 服务可用
- 旧接口兼容别名仍可调用
- joint 名冲突、缺配置、缺 URDF、controller 引用非法 joint 时有明确错误并拒绝启动
- `can_driver` 的 `recover` 不会直接重新使能或恢复运动
- 故障恢复成功后，必须收到 fresh command 才能重新进入有效运动

建议测试场景：
- 正常启动
- `can_driver` 在 `Armed` 状态下收到控制器命令，不应真正下发运动
- `can_driver` 从 `Running` 调用 `halt` 后，电机停止但通信保持
- `can_driver` 进入 `Faulted` 后调用 `recover`，应回到 `Standby` 而非直接运行
- `can_driver` 恢复后若未收到 fresh command，控制命令保持阻断
- 缺少 `master.dcf`
- 缺少 `can_driver.yaml`
- 两边 joint 同名冲突
- 控制器 joint 配置错误
- 位置关节与速度关节同时控制
- 旧接口兼容验证

#### 9. 风险与非目标
主要风险：
- `can_driver` 当前缺少独立 fault reset 原语，生命周期状态机改造需要同步下沉到协议抽象层
- `can_driver` 当前 ROS API 与硬件逻辑耦合较重，接入统一节点时需要做最小侵入式抽离
- `can_driver` 从“写命令时兜底保护”迁移到“显式生命周期控制”时，需要仔细处理旧命令缓存和恢复后的命令重同步
- 调试接口保留兼容后，需要明确文档边界，避免被误用为正式控制接口

非目标：
- 不在本阶段统一原生 YAML
- 不在本阶段统一底层协议实现与 CiA402 细节
- 不在本阶段把两个包重构成单一底层实现

### can_driver 生命周期状态机落地计划
目标：
- 让 `can_driver` 的对外服务语义与 `Eyou_Canopen_Master` 一致
- 消除 `recover` 直接使能的危险行为
- 让统一外观层不再承担“翻译危险旧语义”的责任

实现步骤：
1. 抽出 `can_driver` 侧生命周期协调器，接口命名尽量与 `Eyou_Canopen_Master::OperationalCoordinator` 保持一致。
2. 将 `CanDriverHW` 中 `active_` 的单布尔运行态升级为显式状态枚举，并让 `write()` 根据状态决定是否允许运动命令下发。
3. 在协议层新增 `ResetFault()`，将“故障恢复”和“使能”拆开。
4. 将现有 service 重构为：
   - `init` 负责建链和启动检查
   - `enable` 负责进入 `Armed`
   - `resume` 负责进入 `Running`
   - `halt` 负责 `Running -> Armed`
   - `disable` 负责回到 `Standby`
   - `recover` 负责 `Faulted -> Standby`
   - `shutdown` 负责停通信
5. 增加 command sync / recovery latch：
   - `recover`
   - `enable`
   - `resume`
   - 设备掉线恢复
   以上路径都必须要求 fresh command 后才允许运动。
6. 最后再把统一外观层接到两边 lifecycle coordinator 之上，而不是在 facade 层内做临时语义修补。

代码落点建议：
- `can_driver/include/can_driver/CanProtocol.h`
- `can_driver/include/can_driver/CanDriverHW.h`
- `can_driver/src/CanDriverHW.cpp`
- `can_driver/src/MtCan.cpp`
- `can_driver/src/EyouCan.cpp`
- 如有必要，新增 `can_driver/include/can_driver/operational_coordinator.hpp`
- 如有必要，新增 `can_driver/src/operational_coordinator.cpp`
