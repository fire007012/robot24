# 达妙履带电机接入与拆分实施计划

日期：2026-04-18

实施状态更新：2026-04-19 已将旧 `control` 包降级为 archive-only 参考包；正式执行链固定为 `mobility_control/base_cmd_node -> wheel_controller -> can_driver::DamiaoCan`。

## 1. 目标

将当前新增的 `control` 包拆分为两部分，并纳入现有正式执行链：

- 达妙电机驱动后端进入 `can_driver`
- 底盘北向控制节点进入 `mobility_control`
- 正式执行与生命周期 authority 继续统一收口到 `Eyou_ROS1_Master/hybrid_motor_hw_node`

本次明确约束：

- 达妙电机仅作为履带/底盘驱动电机使用
- 第一阶段只支持 `velocity` 控制语义
- 不在第一阶段接入达妙位置控制、MIT 位置环或力矩控制
- 后续会强依赖 wheel joint，因此需在计划中包含 URDF 与控制器命名对齐

## 2. 当前问题

当前 `control` 包中的 [damiao_diff_chassis_node.cpp](/home/dianhua/Robot24_catkin_ws/src/control/src/damiao_diff_chassis_node.cpp) 同时承担了两层职责：

- 协议/驱动层：
  - SocketCAN 打开与关闭
  - 模式寄存器写入
  - 达妙使能/失能
  - 达妙速度/MIT 报文编码
  - 回执与反馈等待
- 底盘控制层：
  - 订阅 `/cmd_vel`
  - 差速逆运动学
  - 指令超时停车
  - 两轮方向修正

这与当前正式结构不一致：

- `can_driver` 已是统一执行后端
- `mobility_control` 是底盘控制包
- `Eyou_ROS1_Master` 是正式生命周期和共享 `controller_manager` 入口

## 3. 目标架构

目标架构收敛为：

```text
/cmd_vel
  -> mobility_control/base_cmd_node
    -> diff_drive_controller
      -> controller_manager
        -> HybridRobotHW / CanDriverHW
          -> DamiaoCan
            -> SocketCAN(canX)
              -> 达妙履带电机
```

生命周期主链保持：

```text
HybridServiceGateway
  -> HybridOperationalCoordinator
    -> can_driver::OperationalCoordinator
      -> LifecycleDriverOps
        -> DeviceManager
          -> DamiaoCan
```

## 4. 分层边界

## 5. 第一阶段设计约束

### 5.1 达妙只走速度语义

第一阶段在 `can_driver` 里新增 `DamiaoCan` 时，仅支持：

- `control_mode: velocity`
- `CanProtocol::setVelocity()`
- `Enable()`
- `Disable()`
- `Stop()`
- 必要的状态查询/反馈保持

第一阶段不支持：

- `setPosition()`
- `quickSetPosition()`
- `CanProtocol::MotorMode::Position`
- `CanProtocol::MotorMode::CSP`

解析层会显式拒绝 `protocol: DM` 且 `control_mode != velocity` 的配置。

### 5.2 达妙反馈 freshness 不能依赖运动命令

达妙接入 `can_driver` 后，必须像现有 `MT/PP` 一样纳入 `DeviceRefreshWorker`。

原因：

- `can_driver` 在 `Armed` 态不发送运动命令
- `LifecycleDriverOps::enableHealthy()` 和 `motionHealthy()` 依赖 fresh feedback
- 如果不加入 refresh/keepalive，DM 轴会在 `Armed` 态掉 freshness，导致生命周期自动 fault

因此：

- 不能仅实现“写命令”
- 必须实现“状态刷新或空闲保活”

### 5.3 URDF/wheel joint 将成为正式依赖

当前仓库中 wheel/track 相关命名存在历史分叉：

- 文档侧长期使用 `left_track/right_track`
- 当前 [car_urdf.urdf](/home/dianhua/Robot24_catkin_ws/src/car_urdf/urdf/car_urdf.urdf) 中可见四个 wheel joint

本计划要求后续冻结一套正式 wheel joint 命名，并让以下几处完全对齐：

- URDF
- `can_driver` joints YAML
- `diff_drive_controller` 配置
- `mobility_control`
- 依赖 wheel joint 的上层组件

是否最终命名为 `left_track/right_track` 或新的 `left_wheel_joint/right_wheel_joint`，应在 URDF 提交中一次性收口，不再长期并存。
