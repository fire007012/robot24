# 摆臂控制器模式与冷切换设计（2026-04-15）

## 1. 文档目的

本文用于明确四摆臂控制链的正式设计方向，满足以下目标：

- 摆臂控制器与机械臂控制器解耦
- 同时支持 `CSP` 与 `CSV`
- 未来可扩展到 `CST`
- 支持不重启主节点的冷切换
- 北向尽量使用标准 `ros_control` / `controller_manager` 接口

本文基于当前工作区现有实现给出设计结论与后续改造建议。

---

## 2. 当前已确认的系统约束

### 2.1 生命周期 authority 不在摆臂包

当前正式电机执行与生命周期 authority 仍然是：

- `Eyou_ROS1_Master/hybrid_motor_hw_node`

因此摆臂控制链不能自行接管：

- 电机生命周期
- 统一 `controller_manager`
- 硬件主循环

摆臂相关节点只负责：

- 模式管理
- 控制器切换
- 摆臂目标生成
- 摆臂安全约束

### 2.2 模式切换当前只能在 Standby 进行

当前 `Eyou_Canopen_Master` 的 `set_mode` 只允许在 `Standby` 下调用。

这意味着摆臂的 `CSP/CSV/CST` 切换天然应设计为冷切换，而不是运行态热切换。

### 2.3 当前 ROS 控制面只正式暴露了位置与速度接口

当前 `CanopenRobotHwRos` 已注册：

- `JointStateInterface`
- `PositionJointInterface`
- `VelocityJointInterface`

但尚未注册：

- `EffortJointInterface`

并且当前 `write()` 路径只对 `CSV` 走速度命令、对 `IP/CSP` 走位置命令，尚未把 `CST` 通过 `ros_control` effort 命令面真正暴露出来。

因此当前状态应理解为：

- `CSP`：可落地
- `CSV`：可落地
- `CST`：底层内部已有模式定义与力矩通道基础，但尚未形成完整 ROS 标准控制链

---

## 3. 关节命名建议

当前仓库内摆臂存在两套历史命名：

- 旧文档命名：`flipper_1 ~ flipper_4`
- 当前 URDF 命名：`left_front_arm_joint`、`right_front_arm_joint`、`left_rear_arm_joint`、`right_rear_arm_joint`

正式实现时建议以 **URDF 关节名** 为唯一标准名：

- `left_front_arm_joint`
- `right_front_arm_joint`
- `left_rear_arm_joint`
- `right_rear_arm_joint`

原因：

- `ros_control` controller 配置必须与 URDF 完全对齐
- MoveIt / RViz / joint_state / limit 体系都天然依赖 URDF 关节名
- 可以避免后续再维护一层 `flipper_x -> URDF joint` 名称映射

若历史脚本仍使用 `flipper_1~4`，建议只在迁移期保留别名映射，不再作为正式命名继续扩散。

## 4. 总体设计结论

### 4.1 分开“硬件模式”和“北向控制剖面”

摆臂这里需要明确区分两层概念：

#### 第一层：真实硬件模式

这是实际写到驱动器上的模式：

- `CSP`
- `CSV`
- `CST`（未来）

这一层直接对应底层 `set_joint_mode` / `set_mode`，切换时需要遵守当前系统约束：

- 只能在 `Standby` 下切换
- 因此 `CSP <-> CSV <-> CST` 属于冷切换

#### 第二层：北向控制剖面

这是提供给上层控制逻辑使用的语义层，不一定与硬件模式一一对应：

- `csp_position`
- `csp_jog`
- `csv_velocity`
- `cst_torque`（未来）

其中：

- `csp_position` 和 `csp_jog` 都运行在 **同一个真实硬件模式 `CSP`** 上
- `csp_jog` 只是一个北向“速度语义门面”，不是新的硬件模式

这层拆分很关键，因为它允许我们：

- 保持底层模式设计清晰
- 在不切硬件模式的前提下提供“基于 CSP 的速度控制语义”
- 避免把北向速度语义误实现成底层 `CSV`

### 4.2 摆臂采用“单独控制器组 + 单独模式管理”

建议把摆臂从机械臂控制器组中完全独立出来，形成一套单独控制器组：

- `flipper_csp_controller`
- `flipper_csv_controller`
- `flipper_cst_controller`（未来）

这组控制器与机械臂的：

- `arm_position_controller`
- `arm_velocity_controller`

完全分离，只共享同一个底层 `controller_manager`。

### 4.3 同一时刻只允许一个“真实执行控制器”处于 active

四摆臂的真实执行控制器应采用“单活跃控制器”原则：

- `CSP` 时只运行 `flipper_csp_controller`
- `CSV` 时只运行 `flipper_csv_controller`
- `CST` 时只运行 `flipper_cst_controller`

不允许：

- `flipper_csp_controller` 与 `flipper_csv_controller` 同时 active
- `flipper_csv_controller` 与 `flipper_cst_controller` 同时 active
- 摆臂控制器与旧遗留摆臂控制话题同时写同一组关节

这是保证资源占用清晰、命令语义唯一和冷切换可靠的基础。

补充说明：

- `csp_position` 与 `csp_jog` 两个北向控制剖面都可以共享 `flipper_csp_controller`
- 它们之间的切换不属于硬件模式切换，可以在 `flipper_manager_node` 内部完成，不必走冷切换

## 5. 推荐控制器设计

### 5.1 `CSP` 不应直接暴露成原始位置直通接口

当前最需要避免的坑是：

- 上层直接发布“目标角数组”
- 控制器原样转发到 `PositionJointInterface`
- 底层又刚好运行在 `CSP`

这种“位置直通到 CSP”在摆臂实时场景下非常容易造成：

- 抽搐
- 目标突变
- 机械冲击
- 预置位切换不自然

因此：

- 不建议把 `position_controllers/JointGroupPositionController` 作为 `CSP` 的正式北向主入口
- 不建议把“原始位置数组”直接作为 `CSP` 的公开语义接口

### 5.2 `flipper_csp_controller`

建议：

- 类型：`position_controllers/JointTrajectoryController`
- 对应底层模式：`CSP (8)`
- 主要南向接口：`/flipper_csp_controller/command`
- 消息类型：`trajectory_msgs/JointTrajectory`

这里的重点是：

- 虽然它带 `follow_joint_trajectory` action
- 但它**并不要求必须用 action**
- 它本身也支持 `command` topic

因此它可以作为：

- 一个更平滑的 `CSP` 位置参考执行器
- 同时又不强迫上层使用 action

推荐用途：

- 预置位
- 自主翻越段的离散目标
- `csp_jog` 内部生成的连续位置参考

### 5.3 `flipper_csv_controller`

建议：

- 类型：`velocity_controllers/JointTrajectoryController`
- 对应底层模式：`CSV (9)`
- 主要南向接口：`/flipper_csv_controller/command`
- 消息类型：`trajectory_msgs/JointTrajectory`

说明：

- 它仍然可以只用 `command` topic，不必用 action
- 对上层实时控制来说，真实 public API 不一定直接暴露为这个控制器的话题
- 是否由 `flipper_manager_node` 代理后再发到该控制器，由上层架构决定

### 5.4 `flipper_cst_controller`（未来）

建议：

- 类型：`effort_controllers/JointTrajectoryController`
- 对应底层模式：`CST (10)`
- 主要南向接口：`/flipper_cst_controller/command`
- 消息类型：`trajectory_msgs/JointTrajectory`

说明：

- 这一层当前只是未来目标
- 前提是底层补齐 `EffortJointInterface`

### 5.5 `csp_jog`：基于 `CSP` 的北向速度门面

这是当前新增的关键设计。

`csp_jog` 的定义是：

- 北向看起来像“速度控制”
- 南向实际仍然运行在 `CSP`
- 中间由 `flipper_manager_node` 把速度命令变成连续位置参考

它的典型处理链为：

1. 上层发布关节速度意图
2. `flipper_manager_node` 持有内部参考位姿 `q_ref`
3. 按固定周期积分：`q_ref = q_ref + v_cmd * dt`
4. 对 `q_ref` 做关节限位、联动约束、速率钳制
5. 向 `flipper_csp_controller/command` 发布短时域连续位置参考

因此：

- `csp_jog` 是一个 **北向控制剖面**
- 不是一个真实硬件模式
- 也不是一个新的底层 controller type

它的价值在于：

- 维持 `CSP` 的平滑位置跟踪特性
- 同时给上层一个更自然的“速度控制”接口
- 避免为手动实时控制强制切到 `CSV`

### 5.6 `csp_jog` 的实现约束

`csp_jog` 虽然只是一个北向控制剖面，但它本质上承担了一段高频参考生成逻辑，因此实现形态必须按“近实时控制辅助层”来设计。

建议约束如下：

- 第一阶段优先使用 `C++/roscpp` 实现 `flipper_manager_node`
- `Python` 可以继续用于调试工具、HMI 或离线脚本，但不建议承担 `100Hz+` 的核心 `jog` 积分环
- `jog` 参考生成循环建议不低于 `100Hz`，若 CPU 与主循环负载允许，建议目标为 `200Hz`
- 每个周期使用真实 `dt` 积分，并对 `dt` 做上限钳制，避免线程抖动或调度停顿导致 `q_ref` 突跳
- 进入 `csp_jog`、控制器重启、冷切换返回 `CSP` 后，都应先用最新 `joint_states` 重置 `q_ref`
- 速度命令进入积分前，应先做每关节限速、斜率限制或一阶低通，不能把遥控器噪声直接积分成位置目标
- `q_ref` 与实际反馈 `q_meas` 的偏差应被持续监控；若偏差持续超过阈值，应上报 degraded 状态，并在安全时机重同步，而不是无限累计漂移
- 命令超时后的默认策略应是“速度目标归零并保持当前位置参考”，而不是继续沿旧速度积分
- 第一阶段不在执行层引入 `Ruckig`；参考整形只保留可预测、可验证的限速与限斜率机制

## 6. 推荐的标准接口

### 6.1 北向公共接口

建议北向公共接口固定为 `flipper_control` 层，而不是直接把底层 controller topic 暴露为正式上层 API。

推荐公共接口如下：

#### `/flipper_control/command`

- 类型：`trajectory_msgs/JointTrajectory`
- 用途：离散姿态目标、预置位、未来自主翻越段的参考输入
- 默认服务于：
  - `csp_position`
  - 未来也可扩展到 `cst_torque`

#### `/flipper_control/jog_cmd`

- 类型：`control_msgs/JointJog`
- 用途：实时关节速度语义输入
- 默认服务于：
  - `csp_jog`
  - `csv_velocity`

#### `/flipper_control/set_control_profile`

- 类型：模式切换 service
- 用途：切换北向控制剖面

推荐剖面名为：

- `csp_position`
- `csp_jog`
- `csv_velocity`
- `cst_torque`

#### `/flipper_control/state`

- 类型：状态摘要 topic
- 内容建议包含：
  - 当前控制剖面
  - 当前真实硬件模式
  - 是否在切换中
  - 是否 ready
  - 是否命令超时

#### `/flipper_control/active_profile`

- 类型：`std_msgs/String`
- 用途：给 HMI、调试工具和上层状态机提供轻量状态观察面

### 6.2 南向内部接口

建议南向内部依赖以下 controller topic：

- `/flipper_csp_controller/command`
- `/flipper_csv_controller/command`
- `/flipper_cst_controller/command`（未来）

消息类型统一为：

- `trajectory_msgs/JointTrajectory`

说明：

- 这些可以作为 `flipper_manager_node` 的私有南向接口
- 不建议把它们直接承诺为长期稳定的北向业务接口

### 6.3 接口与剖面的映射关系

| 北向控制剖面 | 真实硬件模式 | 南向控制器 | 北向主命令接口 |
|-------------|--------------|------------|----------------|
| `csp_position` | `CSP` | `flipper_csp_controller` | `/flipper_control/command` |
| `csp_jog` | `CSP` | `flipper_csp_controller` | `/flipper_control/jog_cmd` |
| `csv_velocity` | `CSV` | `flipper_csv_controller` | `/flipper_control/jog_cmd` |
| `cst_torque` | `CST` | `flipper_cst_controller` | `/flipper_control/command`（未来） |

### 6.4 `csp_jog` 的接口约束

`/flipper_control/jog_cmd` 推荐使用 `control_msgs/JointJog`，其原因是：

- 它天然表达“关节速度语义”
- 比 `Float64MultiArray` 更自描述
- 比强行把实时 jog 包装成完整 `JointTrajectory` 更贴近用途

对 `csp_jog` 建议增加以下约束：

- 命令超时则自动置零速度
- 进入该剖面时先用当前反馈重置 `q_ref`
- 切换剖面时清空上一个速度命令
- 超限时钳制速度，而不是放大位置误差

### 6.5 反馈接口

统一使用：

- `/joint_states`

不额外定义新的“摆臂状态反馈 topic”作为正式硬件反馈真相源。

若 `flipper_control` 后续需要发布派生状态，可额外发布：

- `/flipper_control/state`
- `/flipper_control/active_profile`

但它们只作为上层摘要，不替代 `/joint_states`。

### 6.6 控制器切换接口

运行时控制器切换统一使用：

- `/controller_manager/switch_controller`

这是标准接口，不建议再发明一套“start_flipper_velocity_controller”之类的私有服务。

### 6.7 真实硬件模式切换接口

关节模式切换统一使用当前已有的：

- `/hybrid_motor_hw_node/set_joint_mode`

对摆臂关节逐个调用。

后续只需在模式映射表中补齐：

- `position -> canopen: 8`
- `velocity -> canopen: 9`
- `torque -> canopen: 10`

即可复用现有统一模式路由能力。

## 7. 冷切换设计

### 7.1 设计原则

摆臂冷切换定义为：

- 不重启 `hybrid_motor_hw_node`
- 不重启 CANopen 主站
- 通过生命周期降级、停止控制器、切模式、再恢复运行实现模式变更

### 7.2 哪些切换需要冷切换

只有涉及 **真实硬件模式变化** 的切换，才需要冷切换。

也就是说：

- `csp_position <-> csp_jog`：**不需要**冷切换
- `csp_position/csp_jog <-> csv_velocity`：需要冷切换
- `csv_velocity <-> cst_torque`：需要冷切换
- `csp_position/csp_jog <-> cst_torque`：需要冷切换

### 7.3 推荐冷切换状态机

建议把冷切换明确实现成一个有限状态机，而不是若干 service 调用的临时串接逻辑。

推荐最小状态如下：

| 状态 | 作用 | 建议超时 |
|------|------|----------|
| `IDLE` | 正常待命，无切换进行中 | 无 |
| `HALTING` | 若当前在 `Running`，先收敛到 `Armed` | `2.0s` |
| `DISABLING` | 进入 `Standby`，满足 `set_mode` 前提 | `1.0s` |
| `STOPPING_CONTROLLER` | 停止当前活跃摆臂控制器 | `0.5s` |
| `SWITCHING_MODE` | 对 4 个摆臂关节执行真实模式切换 | `3.0s` |
| `STARTING_CONTROLLER` | 启动目标模式对应控制器 | `0.5s` |
| `PRIMING_OUTPUT` | 注入一帧安全初值并校验命令面 ready | `0.5s` |
| `ENABLING` | 从 `Standby` 回到 `Armed` | `1.0s` |
| `RESUMING` | 从 `Armed` 回到 `Running` | `1.0s` |
| `DONE` | 切换完成，恢复对外命令 | 无 |
| `ERROR` | 任一步失败或超时后的收敛态 | 无 |

补充约束：

- 每一步都应采用“调用一次 -> 检查返回结果 -> 决定进入下一状态”的模式
- 不建议在状态机内部做盲重试；若确需自动重试，应以“整次切换任务”为单位，次数有限且默认关闭
- 若 4 个关节中存在部分模式切换失败，系统应停留在 `Standby/ERROR`，并明确报告失败关节，不能继续 `enable -> resume`
- 冷切换期间对外状态必须明确标记 `switching=true`

### 7.4 推荐冷切换序列

以 `CSP -> CSV` 为例，推荐序列如下：

1. 若当前在 `Running`，调用 `halt`
2. 调用 `disable`，进入 `Standby`
3. 停止当前活跃摆臂控制器
4. 对 4 个摆臂关节逐个调用 `set_joint_mode`
5. 启动目标模式对应控制器
6. 向新控制器注入安全初值
7. 若切换前系统处于 `Running`，则执行 `enable -> resume`
8. 恢复对外接受摆臂命令

### 7.5 `CSP` 内部剖面切换

以 `csp_position -> csp_jog` 为例，推荐序列如下：

1. 停止接受旧的上层位置目标
2. 读取当前 `joint_states`
3. 用当前反馈重置 `q_ref`
4. 清空旧速度命令
5. 开启 `jog_cmd` 消费

整个过程：

- 不需要 `halt`
- 不需要 `disable`
- 不需要切真实硬件模式
- 不需要切底层 controller

### 7.6 安全初值约定

控制器切换后，必须先写入一帧安全初值，再允许上层新命令透传：

- 切到 `CSP`：写入当前反馈角度
- 切到 `CSV`：写入全零速度
- 切到 `CST`：写入全零力矩

这样可以避免：

- 旧控制器残留命令被新模式误执行
- 切换瞬间摆臂跳变

### 7.7 切换期间的命令门控

冷切换期间建议 `flipper_control` 拒绝接收新摆臂命令，或将其缓存但不立即下发。

不建议切换过程中仍允许遥控器/自主控制并发写命令。

## 8. 节点设计建议

### 8.1 `flipper_control` 包内正式节点

当前正式节点为：

- `flipper_manager_node`

该节点不是电机执行节点，而是摆臂上层管理节点。

### 8.2 `flipper_manager_node` 职责

负责：

- 管理摆臂当前控制剖面
- 管理真实硬件模式与控制剖面之间的映射
- 执行冷切换流程
- 执行 `csp_position <-> csp_jog` 这类非冷切换的内部剖面切换
- 对外提供预置位、联动、限幅、速率约束
- 把高层输入转成标准控制器命令

不负责：

- 维护电机生命周期 authority
- 替代 `controller_manager`
- 直接写底层主站共享内存

### 8.3 `flipper_manager_node` 南向依赖

建议南向只依赖以下标准/既有接口：

- `/controller_manager/switch_controller`
- `/hybrid_motor_hw_node/halt`
- `/hybrid_motor_hw_node/disable`
- `/hybrid_motor_hw_node/enable`
- `/hybrid_motor_hw_node/resume`
- `/hybrid_motor_hw_node/set_joint_mode`
- `/joint_states`
- `/flipper_csp_controller/command`
- `/flipper_csv_controller/command`
- `/flipper_cst_controller/command`（未来）

### 8.4 `flipper_manager_node` 北向接口建议

北向建议分为两层：

#### 标准消息入口

- `trajectory_msgs/JointTrajectory`
- `control_msgs/JointJog`

#### 管理入口

由 `flipper_manager_node` 额外提供：

- 控制剖面切换 service
- 预置位调用 service / topic
- 单侧/同侧/前后联动控制 topic
- 状态摘要 topic

也就是说：

- 命令消息尽量标准
- 管理语义允许少量定制接口

### 8.5 `flipper_manager_node` 的内部执行模型

为了同时满足模式管理与 `csp_jog` 高频参考生成，建议把 `flipper_manager_node` 内部分成至少三类逻辑：

- 管理平面：处理剖面切换、冷切换状态机、预置位请求与参数更新
- 高频控制平面：固定周期处理 `JointJog -> q_ref` 积分、联动展开、限位与下发
- 状态发布平面：周期发布 `/flipper_control/state` 与轻量调试信息

工程建议如下：

- 高频控制平面不应被 service 回调直接阻塞
- 命令缓存应尽量简单，优先保证“最新命令覆盖旧命令”的单一真相
- 关节限位、联动约束、超时门控都应在下发 controller 之前统一生效
- 若后续需要进一步提高确定性，可把高频积分逻辑继续下沉为独立类库，但不新增新的电机执行节点

### 8.6 联动控制的抽象位置

联动控制应放在 `flipper_manager_node` 内部，而不是塞进底层 controller 或 CANopen 驱动层。

推荐做法是：

- 北向仍以标准四关节命令为基础真相接口
- 另外提供一个“联动模式”管理量，由 `flipper_manager_node` 在进入执行器前把低维输入展开成 4 个关节参考

推荐最小联动模式集合：

- `independent`：四摆臂独立
- `left_right_mirror`：按前/后两组展开，前组为 `left_front + right_front`，后组为 `left_rear + right_rear`
- `front_rear_sync`：四摆臂整组同步，单命令展开为四关节同向参考
- `side_pair`：按左右两侧展开，左侧为 `left_front + left_rear`，右侧为 `right_front + right_rear`
- `diagonal_pair`：对角联动
- `custom_matrix`：保留未来扩展到自定义映射矩阵

当前实现补充说明：

- 第一阶段联动展开采用“同号复制”策略，不在 `flipper_manager_node` 内隐式做符号翻转
- 若后续实机确认某些联动需要镜像符号反向，应增加可配置符号矩阵，而不是把符号约定写死到上层 API

联动的处理顺序建议固定为：

1. 接收上层意图
2. 根据联动模式展开为 4 关节目标
3. 再做每关节限位、速率限制与安全约束
4. 最后再下发到对应执行控制器

补充约束：

- 联动逻辑不能覆盖或绕开单关节硬限位
- 任意时刻都应允许退回 `independent`
- 联动模式切换不应隐式触发真实硬件模式切换

## 13. 故障处理矩阵

| 故障场景 | 最小检测条件 | 推荐处理 |
|---------|--------------|----------|
| `JointJog` 命令超时 | 超过配置超时未收到新命令 | 将速度目标置零，保持当前位置参考，并上报 timeout |
| 冷切换某一步超时 | 状态机阶段超过超时阈值 | 进入 `ERROR`，停止恢复运行，等待人工或上层重新发起 |
| 某个关节模式切换失败 | `set_joint_mode` 返回失败 | 保持 `Standby`，报告失败关节，不继续 `enable -> resume` |
| 控制器切换失败 | `switch_controller` 返回失败 | 进入 `ERROR`，禁止对外透传命令 |
| 联动展开后关节超限 | 目标超出单关节限制 | 先做钳制或拒绝该命令，并记录告警 |
| 反馈与参考长期偏离 | `q_ref` 与 `q_meas` 偏差持续超阈值 | 上报 degraded，必要时要求退回 `csp_position` 或重新同步 |

## 14. 不建议的设计

当前不建议采用以下设计：

### 14.1 不建议把摆臂塞进机械臂控制器

原因：

- 模式需求不同
- 生命周期切换频率不同
- 机械臂和摆臂的高层语义不同
- 会让 MoveIt 与翻越策略互相耦合

### 14.2 不建议四个摆臂各做一个模式控制器组

例如每个摆臂单独做：

- 一个位置控制器
- 一个速度控制器
- 一个力矩控制器

这样控制器数量会膨胀，而且模式切换时要管理 12 个控制器，复杂度没有必要。

更合理的是：

- 四摆臂作为一组
- 每种真实硬件模式一个执行控制器

### 14.3 不建议把 `CSP` 北向接口做成原始位置直通

不建议：

- 上层直接发绝对角数组
- 由 forward position controller 原样转发到 `PositionJointInterface`
- 再让底层 `CSP` 自己兜底

这样非常容易出现实时控制抽搐。

### 14.4 不建议把底层 controller topic 直接承诺为北向长期接口

更推荐：

- 北向只暴露 `flipper_control` 层接口
- 底层 controller topic 作为内部实现细节

这样后续如果从 `CSP` 的内部实现切到更平滑的参考生成方式，不需要破坏上层 API。

### 14.5 不建议在第一阶段就把摆臂轨迹 action 做成主入口

第一阶段的主要需求是：

- 遥操作
- 预置位
- 翻越策略
- 模式切换

此时优先做 `command` topic + `flipper_manager_node` 代理更直接。

若后续出现明确的“摆臂时序轨迹执行”需求，再加 trajectory controller 更合适。

## 15. 最终推荐方案

当前最推荐的正式方案是：

1. 摆臂在 `flipper_control` 中单独建设控制链
2. 使用一组独立于机械臂的标准 `ros_control` 执行控制器：
   - `flipper_csp_controller`
   - `flipper_csv_controller`
   - `flipper_cst_controller`（未来）
3. 北向再抽象出控制剖面：
   - `csp_position`
   - `csp_jog`
   - `csv_velocity`
   - `cst_torque`（未来）
4. 第一阶段先落地：
   - `flipper_csp_controller`
   - `flipper_csv_controller`
   - `csp_jog` 北向速度门面
5. `csp_position <-> csp_jog` 不走冷切换
6. `CSP <-> CSV <-> CST` 才走冷切换
7. 冷切换流程由 `flipper_manager_node` 负责编排
8. 生命周期 authority 仍保持在 `Eyou_ROS1_Master/hybrid_motor_hw_node`

一句话总结：

四摆臂应当是“真实硬件模式与北向控制剖面分层设计”，其中 `csp_jog` 作为只存在于北向的一层，为实时控制提供速度语义；底层仍保持 `CSP/CSV/CST` 清晰、可冷切换，而不是继续挂在机械臂控制器后面或再造一层电机执行节点。
