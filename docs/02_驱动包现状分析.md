# 驱动包现状分析

## 1. 背景

系统中并存两个驱动包，分别负责两条 CAN 总线上的电机通讯。
两者都能独立工作，但设计风格和接口形式不同。
本文档记录两者的现状、能力和接口差异，作为后续统一接口设计的基础。

---

## 2. ros_canopen（yiyou_canopen 配置包）

### 基本信息

| 项目 | 内容 |
|------|------|
| 包路径 | `yiyou_canopen` |
| 框架 | 标准 ros_canopen |
| EDS 文件 | `YiyouServo_V1.4.eds`（原始未改版，未改 RPDO 映射） |
| 配置文件 | `bus.yaml`、`motor.yaml`、`yiyou_canopen.launch` |
| 启动节点 | `canopen_motor_node` |
| 状态 | 已配置完成，catkin_make 编译通过 |

### 接口

| 接口 | 类型 | 说明 |
|------|------|------|
| `/canopen/driver/init` | Service | 初始化 |
| `/canopen/driver/recover` | Service | 恢复 |
| `/canopen/driver/get_state` | Service | 状态查询 |
| `/canopen/joint_states` | Topic (JointState) | 关节状态反馈 |
| `/canopen/joint1/command` ~ `/joint6/command` | Topic (Float64, rad) | 单轴位置命令 |

### 架构特点

- `canopen_motor_node` 是独立节点，内部自己创建 `controller_manager` + `RobotLayer`
  （`RobotLayer` 继承自 `hardware_interface::RobotHW`）
- 包含完整的 CiA 402 状态机管理（NMT、6040/6041 握手）
- 位置单位自动换算为 rad

## 3. can_driver（自研包）

### 基本信息

| 项目 | 内容 |
|------|------|
| 包路径 | `can_driver` |
| 框架 | 自行实现 RobotHW + controller_manager |
| 配置文件 | `can_driver.yaml`、`ros_controllers.yaml`、`can_driver.launch` |
| 启动节点 | `can_driver_node` |
| 支持协议 | MT（多圈）/ PP（私有），混合支持 |

### 核心实现（CanDriverHW）

- 继承 `hardware_interface::RobotHW`
- 标准 `read() → cm.update() → write()` 主循环，100Hz
- 按关节 `controlMode` 分别注册 `PositionJointInterface` / `VelocityJointInterface`
- 内置 `positionScale` / `velocityScale` 做 counts ↔ rad 双向换算
- 从 URDF + rosparam 加载关节限位，通过 `joint_limits_interface` 做 saturation
- `DeviceManager` 抽象封装 CAN 设备 / 协议 / 互斥锁
- 支持 direct topic 命令通道（带超时仲裁）+ `debugBypassRosControl_` 旁路标志

### 接口

| 接口 | 类型 | 说明 |
|------|------|------|
| `/can_driver_node/init` | Service | 初始化 CAN 设备 |
| `/can_driver_node/shutdown` | Service | 关闭 |
| `/can_driver_node/recover` | Service | 恢复/使能 |
| `/can_driver_node/motor_command` | Service | 电机命令（使能/禁用/停止/设模式） |
| `/can_driver_node/motor_states` | Topic (MotorState) | 电机状态 |
| `/can_driver_node/motor/<joint>/cmd_position` | Topic (Float64) | 直连位置命令 |
| `/can_driver_node/motor/<joint>/cmd_velocity` | Topic (Float64) | 直连速度命令 |

### write() 中的命令仲裁逻辑

`write()` 同时支持两个命令来源：

1. **ros_control 通道**：`posCmd` / `velCmd`（由 controller 写入）
2. **direct topic 通道**：`directPosCmd` / `directVelCmd`（由外部 topic 写入）

仲裁规则：
- direct topic 命令在 `directCmdTimeoutSec_` 内有效时优先
- 超时后回退到 ros_control 命令
- `debugBypassRosControl_` = true 时完全忽略 ros_control 命令

**注意**：MoveIt 运行期间必须确保 `debugBypassRosControl_` = false，
且无外部程序往 direct topic 发送命令，否则会抢占 JointTrajectoryController 的控制权。

## 4. 两者对比

| 维度 | ros_canopen (yiyou_canopen) | can_driver |
|------|----------------------------|------------|
| HW 抽象 | ros_canopen 内置 RobotLayer | 自行实现 CanDriverHW |
| controller_manager | 节点内嵌 | main.cpp 显式创建 |
| 位置单位（北向） | rad（自动换算） | rad（positionScale 换算） |
| CiA 402 状态机 | ros_canopen 自带完整管理 | 封装在 CanProtocol / DeviceManager |
| 关节限位 | 依赖 URDF / EDS | URDF + rosparam + joint_limits_interface |
| FollowJointTrajectory | 需额外加载 JointTrajectoryController | 框架已就绪，需配置加载 |
| 直连命令通道 | `/jointN/command` topic | `/motor/<joint>/cmd_position` topic + service |
| 支持的控制模式 | 取决于 EDS 配置 | position / velocity 按关节独立配置 |

### 统一接入 MoveIt 的差距

两者默认不是同一种北向接口。MoveIt 想无感切换或混合使用，需要统一到同一北向契约：

- `joint_states`（统一关节命名、单位）
- `FollowJointTrajectory`（JointTrajectoryController）
- 一致的关节限位
