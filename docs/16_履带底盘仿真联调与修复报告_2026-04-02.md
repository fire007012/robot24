# 履带底盘仿真联调与修复报告（2026-04-02）

## 1. 背景与目标

本轮联调针对 `demo_gazebo.launch` 启动链路，重点处理以下几类问题：

1. DS5 真手柄左摇杆左右拨动时，底盘经常只能前进后退，转向无效。
2. Gazebo 中底盘对 `cmd_vel.angular.z` 响应很差，表现与“履带车”语义不一致。
3. MoveIt、Gazebo、`joint_states`、TF 和里程计链路之间存在接口不一致。
4. 需要兼顾后续上实机时的 ROS 接口稳定性，避免仿真和实机使用两套完全不同的话题约定。

本次工作的目标不是只“把车跑起来”，而是把底盘仿真链路梳理成一套可解释、可复现、后续能继续联调的结构。

## 2. 关键结论

### 2.1 左摇杆左右无效，不只是手柄映射问题

联调中确认过：

- 手柄原始输入里，左摇杆横轴是有数据的。
- `ds5_teleop_node` 输出的 `Twist.angular.z` 也是有变化的。

这说明“左摇杆左右没反应”在后期已不再是纯输入层问题，而是底盘仿真执行层没有正确消化角速度指令。

### 2.2 主因在 Gazebo 底盘执行模型，不在 MoveIt

问题的主链路是：

1. 手柄输入正常。
2. ROS `cmd_vel` 已经正常产生。
3. Gazebo 底盘插件对转向指令响应不合理，导致看起来像“左右拨动没效果”。

也就是说，MoveIt 不是这类底盘转向故障的主因。

### 2.3 最终没有采用“joint_states 合并节点”作为长期方案

联调过程中确实讨论过“让 MoveIt 监听带命名空间的 `joint_states`”或“加一个合并节点”的思路，但最终落地方案不是这个。

最终方案是：

- 让 Gazebo 里的轮子关节状态直接发布到根话题 `/joint_states`
- 保持 MoveIt、`robot_state_publisher`、后续实机统一使用同一个标准入口

这样做的原因是：

1. 后续上实机时，标准接口本来就应该是 `/joint_states`。
2. 让仿真也遵守同一接口，比给 MoveIt 单独加命名空间适配更稳。
3. 少一层 topic remap / merge 逻辑，整体链路更简单，也更不容易埋回归问题。

## 3. 根因分析

### 3.1 DS5 转向轴映射存在驱动差异

在当前 Linux `joy` 驱动链路下，DS5 各轴索引和常见 DS4/DS5 示例并不完全一致。实际联调里确认到：

- `axis_lx = 0`
- `axis_rx = 2`
- `axis_ry = 3`
- `axis_l2 = 4`
- `axis_r2 = 5`

此前底盘转向候选轴配置不够稳，导致某些驱动映射下会读错轴，或者兜底逻辑覆盖了预期转向轴。

### 3.2 旧底盘方案更像“四轮滑移”，不是“履带语义”

原链路更接近传统 skid-steer / diff-drive 风格，配合当前模型的接触和摩擦设置后，会出现：

- 前进还可以
- 转向尤其是原地转向表现差
- 用户感知上就是“左摇杆左右没反应”

用户明确要求“履带语义更对”，因此这次没有停留在调小摩擦或继续硬调老插件参数，而是改成更贴近履带语义的插件路线。

### 3.3 `SimpleTrackedVehiclePlugin` 在当前整机模型上不稳定

联调中尝试过 `SimpleTrackedVehiclePlugin`。结果是在当前整机模型上，Gazebo 中机器人模型和部分 link 的 `twist` 很快出现 `NaN`，属于明显不稳定状态。

因此这个方案被判定为“已调查但不适合作为当前项目的落地方案”，没有继续沿这条线推进。

### 3.4 Gazebo tracked 插件与 ROS 标准接口之间存在天然断层

选用 Gazebo tracked 插件后，会出现两个接口问题：

1. 它接收的是 Gazebo transport 上的 `cmd_vel_twist`，不是普通 ROS `Twist` 订阅。
2. 它不会像常见 `gazebo_ros` 轮式插件那样直接给出标准 ROS `odom`/TF。

所以如果不补桥接节点，ROS 侧虽然在发速度，但 Gazebo 底盘不会按预期接收；即便底盘在动，ROS 侧的里程计和 TF 也会断链。

## 4. 已实施修改

### 4.1 DS5 遥控链路修正

修改文件：

- `car_control/scripts/ds5_teleop_node.py`
- `car_control/config/control_params.yaml`

已做内容：

1. 校正 DS5 在当前驱动下的轴索引配置，尤其是 `axis_rx` 从旧映射切到当前实际可用映射。
2. 将底盘转向改为“候选轴优先级”机制，优先使用左摇杆横轴，必要时再用右摇杆或十字键兜底。
3. 保留死区、超时刹停和模式切换机制，避免联调时误动作。

这一步解决的是“输入层是否真的发出了正确转向命令”。

### 4.2 MoveIt / Gazebo 启动链路修正

修改文件：

- `car_moveit_config/launch/demo.launch`
- `car_moveit_config/launch/demo_gazebo.launch`
- `car_moveit_config/launch/gazebo.launch`

已做内容：

1. 给 `demo.launch` 增加 `publish_virtual_joint` 开关。
2. 在 `demo_gazebo.launch` 中关闭虚拟根关节静态 TF，避免 Gazebo 场景下重复发布。
3. 在 `gazebo.launch` 中透传 `gazebo_gui` 到 `empty_world.launch`。
4. 启动时加载 `car_control/config/control_params.yaml`。
5. Gazebo 模式下只拉起机械臂和夹爪 `ros_control` 控制器，不再给底盘额外套一层旧的 base controller。
6. 增加 tracked bridge 和 odom 重建节点。

这一步解决的是“Gazebo 和 MoveIt 谁负责什么、哪些 TF/控制器应该存在”的职责边界问题。

### 4.3 底盘 URDF / Gazebo 插件改造

修改文件：

- `car_urdf/urdf/car_urdf.urdf`

已做内容：

1. 将底盘主碰撞体简化为 box，降低复杂网格碰撞带来的仿真不稳定性。
2. 将四个驱动轮碰撞改为 cylinder，使接触模型更可控。
3. 下调轮关节的 `damping` / `friction`，减少旧设置对转向响应的额外抑制。
4. 增加 Gazebo 轮关节状态发布插件，直接把四个轮关节发到根 `/joint_states`。
5. 放弃旧底盘插件方案，改用 `libWheelTrackedVehiclePlugin.so`。

当前 tracked 插件核心参数为：

- `robot_namespace = car_urdf`
- `left_joint = left_front_wheel_joint, left_rear_wheel_joint`
- `right_joint = right_front_wheel_joint, right_rear_wheel_joint`
- `default_wheel_radius = 0.029`
- `tracks_separation = 0.438`
- `steering_efficiency = 0.3`
- `max_linear_speed = 3.0`
- `max_angular_speed = 3.0`
- `track_mu = 2.0`
- `track_mu2 = 0.5`

这一步是本轮修复的核心。它把底盘从“普通轮式滑移近似”切到了“更接近履带语义的四轮承载式 tracked 近似”。

### 4.4 新增 ROS <-> Gazebo 桥接节点

修改文件：

- `car_control/src/tracked_cmd_bridge_node.cpp`
- `car_control/scripts/gazebo_model_odom_node.py`
- `car_control/CMakeLists.txt`
- `car_control/package.xml`

### `tracked_cmd_bridge_node`

作用：

- 订阅 ROS 侧 `/car_urdf/cmd_vel`
- 转发到 Gazebo transport 侧 `~/car_urdf/cmd_vel_twist`
- 对速度做限幅、符号缩放和超时刹停

说明：

- 当前 `linear_scale = -1.0`
- 当前 `angular_scale = -1.0`

这是联调后为匹配当前模型运动方向保留下来的符号修正。

### `gazebo_model_odom_node`

作用：

- 从 `/gazebo/model_states` 提取 `robot` 模型状态
- 发布 `/car_urdf/odom`
- 广播 `odom -> base_link_root`
- 将 Gazebo 世界系平面速度旋回 base frame，供 ROS 下游正常使用

### 构建依赖补充

为保证以上节点可编译、可安装，补充了以下依赖：

- `roscpp`
- `nav_msgs`
- `gazebo_msgs`
- `tf2_ros`

## 5. 为什么最终选择 `WheelTrackedVehiclePlugin`

本轮不是简单“试了个新插件”，而是做过路线取舍。

### 备选路线 A：继续沿用旧轮式方案硬调参数

优点：

- 工程量最小

缺点：

- 很难真正贴近履带车语义
- 用户最关心的“左右拨动为什么像没反应”仍然容易反复出现

### 备选路线 B：改用 `SimpleTrackedVehiclePlugin`

优点：

- 名义上更像“履带”

缺点：

- 在当前整机模型上联调时出现 `NaN`，不稳定
- 当下不能作为可交付方案

### 最终路线：`WheelTrackedVehiclePlugin`

优点：

1. 能利用四个轮接触去近似履带式驱动。
2. 当前模型下可稳定运行。
3. 前进和转向响应已经回到正确量级，能继续联调。

局限：

1. 它仍然不是“完整履带带体接触模型”。
2. 原地转向仍有一定漂移，不应把它等同于高保真履带物理。

### 6.2 仿真联调结果

在隔离 ROS/Gazebo 会话中，对当前代码做过实际测试，结果如下。

#### 前进测试

指令：

- `linear.x = 0.8`
- 持续约 `3s`

结果：

- 平均 `odom.linear.x ≈ 0.792 m/s`
- 平均 `odom.angular.z ≈ 0.025 rad/s`
- 位姿增量约 `dx ≈ 2.168 m`
- 位姿增量约 `dy ≈ -0.073 m`

结论：

- 前进响应已基本正常，速度量级正确。

#### 转向测试

指令：

- `angular.z = 1.5`
- 持续约 `3s`

结果：

- 平均 `odom.angular.z ≈ 1.448 rad/s`
- 平均 `odom.linear.x ≈ -0.007 m/s`
- 仍存在转向时平移漂移：
  - `dx ≈ 0.241 m`
  - `dy ≈ -0.120 m`

结论：

- 转向能力已经恢复到可用状态，不再是“几乎不转”。
- 但还没有达到理想的“严格原地转”。

## 7. 当前结构

经过本轮调整后，`demo_gazebo.launch` 下的底盘链路可以概括为：

1. DS5 手柄发布 `/joy`
2. `ds5_teleop_node` 生成 `/car_control/cmd_vel`
3. `base_cmd_node` 做限幅和超时保护，输出 `/car_urdf/cmd_vel`
4. `tracked_cmd_bridge_node` 把 ROS `Twist` 转成 Gazebo transport `~/car_urdf/cmd_vel_twist`
5. `WheelTrackedVehiclePlugin` 驱动 Gazebo 底盘运动
6. `gazebo_model_odom_node` 从 `/gazebo/model_states` 重建 `/car_urdf/odom` 和 `odom -> base_link_root`
7. Gazebo 轮关节状态直接发布到根 `/joint_states`
8. MoveIt 和 `robot_state_publisher` 继续按标准 ROS 接口消费 `/joint_states`

这套结构的特点是：

- Gazebo 底盘内部允许使用 tracked 插件自己的实现方式
- 对 ROS 上层仍然保持标准接口，不把特殊性扩散到 MoveIt 侧

## 8. 遗留问题

目前仍有以下已知问题未完全消除：

1. 原地转向时仍存在一定平移漂移，说明 tracked 近似还不够“硬”。
2. `WheelTrackedVehiclePlugin` 只是近似履带模型，不是显式履带带体接触模型。
3. 联调过程中仍观察到与 `zhua1` / `zhua2` 相关的 `TF_NAN_INPUT` 噪声，这一项不属于本轮底盘修复范围。

## 9. 后续建议

### 9.1 如果目标是“能联调、能继续开发”

当前方案已经足够继续推进：

- 手柄到底盘的控制链路已打通
- MoveIt 与 Gazebo 的接口已经统一
- 前进/转向响应已恢复到正确量级

### 9.2 如果目标是“更强履带真实性”

后续可以继续做两类深入工作：

1. 进一步细调 tracked 插件参数与接触参数，继续压低转向漂移。
2. 如果必须追求更高保真履带物理，再单独评估更重的路线：
   - 继续清理模型/碰撞后重新挑战 `SimpleTrackedVehiclePlugin`
   - 或引入更专门的履带接触建模方案

但这已经超出本轮“先修通联调链路”的范围。
