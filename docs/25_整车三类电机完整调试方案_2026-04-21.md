# 整车三类电机完整调试方案（MT 底盘 + ECB 摆臂 + 机械臂全轴）

本文给出一套可直接执行的联调流程，覆盖：

- 主电机 MT（前轮/底盘移动）
- ECB 网络电机（摆臂电机）
- 机械臂六轴 + 夹爪（单电机控制开合）

目标是让你在实机上按阶段排错，避免一次性全开导致故障来源不清。

---

## 1. 调试边界与当前事实

当前工程中关键入口：

- 统一硬件门面：`Eyou_ROS1_Master/launch/hybrid_motor_hw.launch`
- 顶层整机启动：`robot_bringup/launch/full_system.launch`
- MT 调试接口：`can_driver/scripts/mt_motor_interface.py`
- ECB 发现脚本：`can_driver/scripts/discover_ecb_and_enable_yaml.sh`
- ECB 联调脚本：
  - `can_driver/scripts/test_ecb_motor_motion.sh`
  - `can_driver/scripts/test_ecb_group_motion.sh`
  - `can_driver/scripts/test_ecb_four_motors.sh`
- 摆臂调试 UI：`flipper_control/scripts/flipper_motor_debug_ui.py`
- 机械臂 Action 入口：`arm_control/scripts/arm_goal_executor_node.py`
- 夹爪节点：`arm_control/scripts/gripper_cmd_node.py`

当前配置要点：

- 机械臂为 6 轴（`shoulder_*` 到 `wrist_yaw_joint`），已纳入 `main_arm`。
- 夹爪按单电机控制：`left_gripper_finger_joint`（右指 mimic）。
- `can1` 当前统一承载机械臂全轴：前 2 轴 MT（`shoulder_yaw_joint`、`shoulder_pitch_joint`）+ 后 4 轴 PP，统一走 `can_driver`。
- `can0` 当前承载底盘前两轮 MT 电机：`left_front_wheel_joint`、`right_front_wheel_joint`。
- 摆臂 4 轴当前不走 `can0/can1` SocketCAN，也不走 CANopen，而是走 `can_driver` 的 ECB 网络设备（`ecb://...`）；若未来硬件版本要求 ECB 侧也挂 CAN，则归到 can0。

## 2. 安全前置（必须执行）

1. 机械安全

- 先让车体离地或低速空载。
- 摆臂和机械臂周围清空人员。
- 急停可用，电源可快速断开。

2. 软件环境

```bash
cd /home/fire/catkin_ws
source /opt/ros/noetic/setup.bash
source devel/setup.bash
```

3. 通讯链路

```bash
ip -4 -brief addr
ip link
```

- MT/PP 依赖 `can0/can1` 等 SocketCAN。
- ECB 依赖以太网网段（常见 `192.168.1.x`）。

当前实机分工建议按下面理解，避免把摆臂误接到 CANopen 侧：

- `can0`：SocketCAN + `can_driver`，底盘前两轮 MT 电机。
- `can1`：SocketCAN + `can_driver`，机械臂全轴。
- `ecb://...`：摆臂 4 轴，由 `can_driver` 通过 ECB 网络控制；默认不占用 can0/can1。

## 3. 统一基线启动（所有阶段共用）

建议先起统一门面，不自动上电：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  auto_init:=false auto_enable:=false auto_release:=false \
  enable_ecb_control:=false optional_ecb_init:=true primary_can_device:=''
```

说明：当前 hybrid 已支持 can_driver 同时保留 `can1` + `ecb://...` 两类 joints，
并通过 launch 参数决定 ECB 是否参与生命周期：

其中 CANopen 与 can_driver 的边界是：

- 当前实机默认不启用 CANopen 后端；机械臂全轴已改由 `can_driver` 托管，其中前 2 轴为 MT、后 4 轴为 PP。
- can_driver 侧负责机械臂全轴、底盘前两轮，以及通过 ECB 网络接入的摆臂 4 轴。
- 摆臂 4 轴虽然物理上不是 SocketCAN，但在 hybrid 中仍由 can_driver 生命周期统一托管。

- `enable_ecb_control:=false`（默认）：跳过 ECB 初始化；若配置里存在 ECB 设备会输出 WARN，但不影响 `can1` 侧运行。
- `enable_ecb_control:=true`：ECB 也参与初始化（与 `can1` 并行兼容）。
- `optional_ecb_init:=true`（默认）：当 ECB 初始化失败时仅告警，不阻塞整体 init。
- `primary_can_device:=''`：留空自动探测主 SocketCAN 设备（推荐）。
- `auto_detect_primary_can_device:=true`（默认）：自动从系统可用 CAN 网卡中选择主设备。
- `auto_remap_single_socketcan_device:=true`（默认）：当配置仅有一个 SocketCAN 设备且与实机口不一致时，自动重映射（例如 can1 -> can0）。

`can_driver_joints_overlay` 仍可选：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  can_driver_joints_overlay:=''
```

若你希望继续使用“只加载 can1 joints”的覆盖层，可显式指定：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  can_driver_joints_overlay:=$(rospack find Eyou_ROS1_Master)/config/can_driver_joints_can1_only.yaml
```

若当前只想联调机械臂 can1 电机，不希望 init 等待底盘轮子反馈，改用机械臂专用覆盖层：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  auto_init:=false auto_enable:=false auto_release:=false \
  enable_ecb_control:=false optional_ecb_init:=true primary_can_device:=can1 \
  start_wheel_controller:=false \
  can_driver_joints_overlay:=$(rospack find Eyou_ROS1_Master)/config/can_driver_joints_arm4_can1_only.yaml
```

此时 startup sync 只要求机械臂 joints 返回反馈；
不会再因 `left_front_wheel_joint/right_front_wheel_joint` 未接入或 `left_gripper_finger_joint` 零点未校准而阻塞 `init`。

若要联调整个机械臂 6 轴，则无需再单独拉起 roscanopen 或 `canopen_hw_node`；
`shoulder_yaw_joint`、`shoulder_pitch_joint` 已在标准 `can_driver.yaml` 中作为 MT 关节由统一门面直接管理。

另开终端执行生命周期：

```bash
rosservice call /hybrid_motor_hw_node/init "{}"
rosservice call /hybrid_motor_hw_node/enable "{}"
rosservice call /hybrid_motor_hw_node/resume "{}"
```

若 `init` 阶段出现等待或轴未就绪，先按下面排查归属是否正确：

- `shoulder_yaw_joint`、`shoulder_pitch_joint` 属于 can1 上的 MT/can_driver。
- `left_front_wheel_joint`、`right_front_wheel_joint` 属于 can0 上的 MT/can_driver。
- 摆臂 4 轴不应出现在 can0/can1 等待列表；若出现，说明现场仍沿用了旧的历史配置，需要切回 ECB 方案。

检查状态：

```bash
rostopic echo -n 1 /hybrid_motor_hw_node/joint_runtime_states
rosservice call /controller_manager/list_controllers
```

## 4. MT 主电机（底盘）调试方案

### 4.1 目标

验证 MT 电机能完成：

- 使能 / 失能
- 速度模式移动（底盘可动）
- 位置模式（MIT）基础动作

### 4.2 配置检查

- `can_driver/config/can_driver.yaml` 中 MT 关节应为 `protocol: MT`。
- `can_driver/config/mt_control_profiles.yaml` 中 profile 和 `mt_motor_ids` 正确。

列出 profile：

```bash
rosrun can_driver mt_motor_interface.py --profile car_a --action list
```

### 4.3 单电机动作（先单轴，再底盘）

```bash
# 使能
rosrun can_driver mt_motor_interface.py --profile car_a --action enable --motor-id 0x141

# 速度模式 + 速度命令
rosrun can_driver mt_motor_interface.py --profile car_a --action mode --motor-id 0x141 --value 1
rosrun can_driver mt_motor_interface.py --profile car_a --action velocity --motor-id 0x141 --value 1.0

# 停止
rosrun can_driver mt_motor_interface.py --profile car_a --action stop --motor-id 0x141
```

MIT 位置模式：

```bash
export CAN_DRIVER_MT_USE_MIT_POSITION=1
rosrun can_driver mt_motor_interface.py --profile car_a --action mode --motor-id 0x141 --value 0
rosrun can_driver mt_motor_interface.py --profile car_a --action position --motor-id 0x141 --value 0.3
```

### 4.4 底盘链路验证

底盘链路：`/cmd_vel -> mobility_control/base_cmd_node -> /wheel_controller/cmd_vel`

```bash
roslaunch mobility_control base_cmd.launch
rostopic pub -1 /cmd_vel geometry_msgs/Twist '{linear: {x: 0.2}, angular: {z: 0.1}}'
rostopic echo /wheel_controller/cmd_vel
```

验收标准：

- `wheel_controller` 为 `running`
- 低速指令下底盘平稳移动，无突跳

## 5. ECB 网络摆臂电机调试方案

> 推荐先做 ECB 底层联调，再进入摆臂上层模式调试。

这里的前提是：摆臂 4 轴当前已从历史 CANopen 方案切换到 ECB 网络方案。
因此本节所有摆臂联调都应落在 `can_driver_node` 和 ECB 发现脚本上，而不是 `can0` 或历史 `canopen_hw_node`。

### 5.1 ECB 网络发现与回填

```bash
cd /home/fire/catkin_ws/src/robot24/can_driver
bash scripts/discover_ecb_and_enable_yaml.sh 1
```

若提示权限问题，执行：

```bash
sudo sh lib/innfos-cpp-sdk/tools/logtool/linux_x86_64/setpath.sh
newgrp pcap
```

### 5.2 单电机 ECB 验证

```bash
cd /home/fire/catkin_ws/src/robot24/can_driver
bash scripts/test_ecb_motor_motion.sh 3 1.0 2.0 1.0 2.0 /can_driver_node
```

关注：

- `/can_driver_node/lifecycle_state` 是否稳定 `Running`
- `/can_driver_node/motor_states` 中 `fault=false`

### 5.3 多电机 ECB 验证（摆臂组）

```bash
cd /home/fire/catkin_ws/src/robot24/can_driver
bash scripts/test_ecb_group_motion.sh 1.0 2.0 1.0 2.0 /can_driver_node
# 或

```

### 5.4 摆臂上层模式调试（flipper_control）

启动：

```bash
roslaunch flipper_control flipper_control.launch
rosrun flipper_control flipper_motor_debug_ui.py \
  --flipper-ns /flipper_control \
  --backend-type auto \
  --hybrid-ns /hybrid_motor_hw_node
```

说明：当前默认只需要 `hybrid` 后端；
摆臂 4 轴的实际控制路径应落在 hybrid -> can_driver -> ECB，而不是 CANopen。

在 UI 内按顺序：

1. `init`
2. `enable`
3. `resume`
4. 切 `csp_position / csp_jog / csv_velocity`
5. 发位置轨迹或速度点动

验收标准：

- `active_profile` 与 `active_controller` 一致
- 四摆臂关节 `online/enabled=true`
- 无持续 `fault/hb_lost`


至少应有：

- `joint_state_controller`（running）
- `arm_position_controller`（running）

## 7. 全链路联调顺序（推荐）

1. 仅起统一门面 + 生命周期（不跑 MoveIt）
2. MT 单轴 -> 底盘 `/cmd_vel` 链路
3. ECB 单轴 -> ECB 分组
4. flipper_control 模式切换与轨迹
5. 机械臂六轴 Action
6. 夹爪单电机开合
7. 最后再起 full_system 做整机回归

整机启动命令：

```bash
roslaunch robot_bringup full_system.launch \
  enable_moveit:=true \
  enable_mobility_control:=true \
  enable_flipper_control:=true \
  enable_gripper_cmd:=true
```

## 8. 常见故障与处理

1. 生命周期卡在 `Faulted`

- 先看 `motor_states` 哪个轴 fault。
- 执行：

```bash
rosservice call /hybrid_motor_hw_node/recover "{}"
rosservice call /hybrid_motor_hw_node/enable "{}"
rosservice call /hybrid_motor_hw_node/resume "{}"
```

2. ECB 发现不到设备

- 检查网口是否真实 `UP` 且在 `192.168.1.x`。
- 执行 logtool 权限脚本并 `newgrp pcap`。

3. 机械臂可规划不可执行

- `arm_position_controller` 未 running，或 joint 集合不一致。
- 检查：

```bash
rosservice call /controller_manager/list_controllers
rostopic echo -n 1 /joint_states
```

4. 夹爪无动作

- 确认 `gripper_controller` 已启动。
- 确认只控 `left_gripper_finger_joint`，右指不应独立下发。
