# 整车三类电机完整调试方案（MT 底盘 + ECB 摆臂 + 机械臂全轴）

本文给出一套可直接执行的联调流程，覆盖：

- 主电机 MT（前轮/底盘移动）
- ECB 网络电机（摆臂电机）
- 机械臂六轴 + 夹爪（单电机控制开合）

目标是让你在实机上按阶段排错，避免一次性全开导致故障来源不清。

---

## 1. 调试边界与当前事实

当前工程中关键入口：

- 统一硬件门面：`Eyou_ROS1_Master/launch/hybrid_motor_hw.launch`
- 顶层整机启动：`robot_bringup/launch/full_system.launch`
- MT 调试接口：`can_driver/scripts/mt_motor_interface.py`
- ECB 发现脚本：`can_driver/scripts/discover_ecb_and_enable_yaml.sh`
- ECB 联调脚本：
  - `can_driver/scripts/test_ecb_motor_motion.sh`
  - `can_driver/scripts/test_ecb_group_motion.sh`
  - `can_driver/scripts/test_ecb_four_motors.sh`
- 摆臂调试 UI：`flipper_control/scripts/flipper_motor_debug_ui.py`
- 机械臂 Action 入口：`arm_control/scripts/arm_goal_executor_node.py`
- 夹爪节点：`arm_control/scripts/gripper_cmd_node.py`

当前配置要点：

- 机械臂为 6 轴（`shoulder_*` 到 `wrist_yaw_joint`），已纳入 `main_arm`。
- 夹爪按单电机控制：`left_gripper_finger_joint`（右指 mimic）。
- 机械臂当前分布为：前 2 轴 MT（`shoulder_yaw_joint`、`shoulder_pitch_joint`）在 `can0`，后 4 轴 PP 与夹爪在 `can1`，统一走 `can_driver`。
- `can0` 当前承载底盘前两轮 MT 电机：`left_front_wheel_joint`、`right_front_wheel_joint`。
- 摆臂 4 轴当前不走 `can0/can1` SocketCAN，也不走 CANopen，而是走 `can_driver` 的 ECB 网络设备（`ecb://...`）；若未来硬件版本要求 ECB 侧也挂 CAN，则归到 can0。
- `robot_bringup/full_system.launch` 已内含 `Eyou_ROS1_Master/hybrid_motor_hw.launch`，并由 unified facade 统一托管 controller_manager；实机默认不应再额外并发拉起独立 `canopen_hw_node`。

本次按工作区内所有相关 launch 与 README 复筛后的结论是：

- 当前默认实机入口只有两类：`Eyou_ROS1_Master/launch/hybrid_motor_hw.launch` 与 `robot_bringup/launch/full_system.launch`。
- 历史 CANopen 单后端入口仍然存在，但仅用于遗留调试链路，不得与 `full_system.launch` 同时运行。
- `flipper_control` 虽支持 `backend_type:=canopen` 兼容模式，但该模式仅能接到独立 `canopen_hw_node`；整车联调时必须保持 `backend_type:=hybrid` 或 `auto` 指向 hybrid。

## 2. 安全前置（必须执行）

1. 机械安全

- 先让车体离地或低速空载。
- 摆臂和机械臂周围清空人员。
- 急停可用，电源可快速断开。

2. 软件环境

```bash
cd /home/fire/catkin_ws
source /opt/ros/noetic/setup.bash
source devel/setup.bash
```

3. 通讯链路

```bash
ip -4 -brief addr
ip link
```

- MT/PP 依赖 `can0/can1` 等 SocketCAN。
- ECB 依赖以太网网段（常见 `192.168.1.x`）。

当前实机分工建议按下面理解，避免把摆臂误接到 CANopen 侧：

- `can0`：SocketCAN + `can_driver`，底盘前两轮 MT 电机 + 机械臂前 2 轴 MT。
- `can1`：SocketCAN + `can_driver`，机械臂后 4 轴 PP 与夹爪。
- `ecb://...`：摆臂 4 轴，由 `can_driver` 通过 ECB 网络控制；默认不占用 can0/can1。

本方案统一约束如下：

- 仅底盘前轮电机 `0x141` 与 `0x142` 使用速度模式测试。
- 其余所有电机，包括机械臂前 2 轴 MT、机械臂后 4 轴 PP、夹爪以及 ECB 摆臂，全部按位置模式测试。

### 2.1 禁止与 full_system.launch 同时启动的清单

以下入口会单独拉起历史 CANopen 后端、独立 controller spawner 或把摆臂上层切到 `/canopen_hw_node`，与整机统一门面并发时会造成重复占用同一批电机、控制器命名空间冲突，或让现场误判故障归属。

禁止并发的 launch：

- `Eyou_Canopen_Master/launch/bringup.launch`
- `Eyou_Canopen_Master/launch/canopen_hw.launch`
- `Eyou_Canopen_Master/launch/bringup_flipper_4axis.launch`
- `Eyou_Canopen_Master/launch/canopen_hw_flipper_4axis.launch`

禁止并发采用的 README/文档命令来源：

- `Eyou_Canopen_Master/README.md` 中所有 `roslaunch Eyou_Canopen_Master ...` 与 `/canopen_hw_node/*` 生命周期命令
- `flipper_control/README.md` 中 `backend_type:=canopen canopen_ns:=/canopen_hw_node` 的遗留兼容示例
- `Eyou_Canopen_Master/docs/usage.md` 与 `docs/command_cheatsheet.md` 中所有直接启动 `/canopen_hw_node` 的示例

换句话说，只要命令满足下面任一特征，就不要与 `full_system.launch` 同时运行：

- 直接启动 `Eyou_Canopen_Master` 的 `bringup.launch` / `canopen_hw.launch`
- 直接启动 `Eyou_Canopen_Master` 的四摆臂 `bringup_flipper_4axis.launch` / `canopen_hw_flipper_4axis.launch`
- 任何命令中出现 `/canopen_hw_node/init`、`/canopen_hw_node/enable`、`/canopen_hw_node/resume`
- 任何 `flipper_control.launch backend_type:=canopen canopen_ns:=/canopen_hw_node`

若现场已经起过上述任一入口，执行 `full_system.launch` 前必须先彻底停掉旧链路，再确认 ROS 图中没有遗留 `canopen_hw_node` 与第二套 controller spawner。

## 3. 统一基线启动（所有阶段共用）

先做一次“还原检查点”，确认没有遗留历史 CANopen 进程或命名空间：

```bash
rosnode list | egrep 'canopen_hw_node|hybrid_motor_hw_node|controller_spawner'
rosservice list | egrep '/canopen_hw_node|/hybrid_motor_hw_node'
```

期望结果：

- 准备走 `hybrid_motor_hw.launch` 单独联调时，只应看到 `/hybrid_motor_hw_node` 相关接口。
- 准备走 `full_system.launch` 时，启动前不应已经存在 `/canopen_hw_node`。
- 若仍能看到 `/canopen_hw_node`，说明现场未回到统一基线，先停止历史 CANopen bringup，再继续。

建议先起统一门面，不自动上电：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  profile_auto_lifecycle:=false \
  auto_init:=false auto_enable:=false auto_release:=false \
  enable_ecb_control:=false optional_ecb_init:=true primary_can_device:=''
```

从 2026-04-25 起，统一门面和整机总启动都支持 `can_interface_profile`，
可直接按接口裁剪启动参与 lifecycle 的 joints，避免“只测一类电机时，其他接口没通也把 init 拖死”。

- `can_interface_profile:=all`：默认，全量 joints。
- `can_interface_profile:=can0`：仅加载 `can0` 关节，适合测试底盘前轮 MT + 机械臂前 2 轴 MT。
- `can_interface_profile:=can1`：仅加载 `can1` 关节，适合测试机械臂后 4 轴 PP + 夹爪。
- `can_interface_profile:=ecb_network`：仅加载 ECB 网络关节，并自动打开 ECB lifecycle 初始化。
- 自 2026-04-26 起，`hybrid_motor_hw.launch` / `full_system.launch` 在以上三个分模块 profile 下默认也会自动执行 `init -> enable -> resume`，不再要求额外手动调用 lifecycle 服务。
- 若现场需要保留人工分阶段 lifecycle 排障，可显式覆盖 `profile_auto_lifecycle:=false auto_init:=false auto_enable:=false auto_release:=false`。

推荐直接使用下面这些命令，而不是手工记忆 overlay 文件名：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  can_interface_profile:=can0 \
  start_wheel_controller:=true enable_ecb_control:=false
```

此模式下 launch 会自动把默认控制器收敛为当前关节集可用的子集；
`can0` 只会启动 `joint_state_controller` 与可选的 `wheel_controller`，
不会再去加载依赖 `elbow_pitch_joint` 或摆臂关节的控制器。

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  can_interface_profile:=can1 \
  start_wheel_controller:=false enable_ecb_control:=false
```

`can1` profile 默认只保留当前关节集可安全加载的控制器；若后续要做 4 轴/夹爪轨迹控制，
应按第 6 节使用机械臂专用 overlay 或显式传入控制器集合。

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  can_interface_profile:=ecb_network
```

`ecb_network` profile 默认仅保留 `joint_state_controller`，避免误加载历史摆臂控制器名。

若要走整机入口，同样透传该参数即可：

```bash
roslaunch robot_bringup full_system.launch \
  can_interface_profile:=can0
```

若你在分模块 profile 下就是想保留人工生命周期操作，则需要显式关掉 profile 默认自动拉起：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  can_interface_profile:=can0 \
  profile_auto_lifecycle:=false \
  auto_init:=false auto_enable:=false auto_release:=false \
  start_wheel_controller:=true enable_ecb_control:=false
```

说明：当前 hybrid 已支持 can_driver 同时保留 `can1` + `ecb://...` 两类 joints，
并通过 launch 参数决定 ECB 是否参与生命周期：

其中 CANopen 与 can_driver 的边界是：

- 当前实机默认不启用 CANopen 后端；机械臂全轴已改由 `can_driver` 托管，其中前 2 轴为 MT、后 4 轴为 PP。
- can_driver 侧负责机械臂全轴、底盘前两轮，以及通过 ECB 网络接入的摆臂 4 轴。
- 摆臂 4 轴虽然物理上不是 SocketCAN，但在 hybrid 中仍由 can_driver 生命周期统一托管。

- `enable_ecb_control:=false`（默认）：跳过 ECB 初始化；若配置里存在 ECB 设备会输出 WARN，但不影响 `can1` 侧运行。
- `enable_ecb_control:=true`：ECB 也参与初始化（与 `can1` 并行兼容）。
- `optional_ecb_init:=true`（默认）：当 ECB 初始化失败时仅告警，不阻塞整体 init。
- `primary_can_device:=''`：留空自动探测主 SocketCAN 设备（推荐）。
- `auto_detect_primary_can_device:=true`（默认）：自动从系统可用 CAN 网卡中选择主设备。
- `auto_remap_single_socketcan_device:=true`（默认）：当配置仅有一个 SocketCAN 设备且与实机口不一致时，自动重映射（例如 can1 -> can0）。

这里再次强调：`hybrid_motor_hw.launch` 默认不会主动启用 CANopen；只有显式传入 `canopen_dcf_path` 与 `canopen_joints_path` 时，才会附加兼容 CANopen 后端。当前整车默认方案不需要这样做。

`can_driver_joints_overlay` 仍可选：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  can_driver_joints_overlay:=''
```

若你希望继续使用“只加载机械臂 joints”的覆盖层，可显式指定：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  can_driver_joints_overlay:=$(rospack find Eyou_ROS1_Master)/config/can_driver_joints_arm_only.yaml
```

若当前只想联调机械臂，不希望 init 等待底盘轮子反馈，改用机械臂后 4 轴专用覆盖层：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  profile_auto_lifecycle:=false \
  auto_init:=false auto_enable:=false auto_release:=false \
  enable_ecb_control:=false optional_ecb_init:=true \
  start_wheel_controller:=false \
  can_driver_joints_overlay:=$(rospack find Eyou_ROS1_Master)/config/can_driver_joints_arm_rear4_only.yaml
```

此时 startup sync 只要求机械臂 joints 返回反馈；
不会再因 `left_front_wheel_joint/right_front_wheel_joint` 未接入或 `left_gripper_finger_joint` 零点未校准而阻塞 `init`。

若要联调整个机械臂 6 轴，则无需再单独拉起 roscanopen 或 `canopen_hw_node`；
`shoulder_yaw_joint`、`shoulder_pitch_joint` 已在标准 `can_driver.yaml` 中作为 MT 关节由统一门面直接管理。

若你是从历史 CANopen 操作手册切换过来，最容易犯的错有两个：

- 先起了 `Eyou_Canopen_Master/bringup.launch`，再起 `full_system.launch`
- 把 `flipper_control.launch` 显式切到 `backend_type:=canopen`

这两种都不属于当前实机基线，会让问题表象变成“同一控制器已存在”“同一关节被两套后端同时接管”或“服务命名空间看起来正常但实际控制链路分叉”。

另开终端执行生命周期：

```bash
rosservice call /hybrid_motor_hw_node/init "{}"
rosservice call /hybrid_motor_hw_node/enable "{}"
rosservice call /hybrid_motor_hw_node/resume "{}"
```

若 `init` 阶段出现等待或轴未就绪，先按下面排查归属是否正确：

- `shoulder_yaw_joint`、`shoulder_pitch_joint` 属于 can0 上的 MT/can_driver。
- `left_front_wheel_joint`、`right_front_wheel_joint` 属于 can0 上的 MT/can_driver。
- 摆臂 4 轴不应出现在 can0/can1 等待列表；若出现，说明现场仍沿用了旧的历史配置，需要切回 ECB 方案。

检查状态：

```bash
rostopic echo -n 1 /hybrid_motor_hw_node/joint_runtime_states
rosservice call /controller_manager/list_controllers
```

## 4. 按总线拆分的测试基线

先按总线理解当前实机电机分布，再执行对应测试。

### 4.1 can0：MT 电机

can0 上当前包含 4 台 MT 电机：

- `left_front_wheel_joint` -> `0x141`，底盘左前轮
- `right_front_wheel_joint` -> `0x142`，底盘右前轮
- `shoulder_yaw_joint` -> `0x143`，机械臂第 1 轴 / 旋转台
- `shoulder_pitch_joint` -> `0x144`，机械臂第 2 轴 / 大臂

说明：

- 底盘前轮默认走 `velocity` 语义。
- 除底盘 `0x141/0x142` 外，其余电机默认都走位置语义。
- 这 4 台电机都由 `can_driver` 托管，物理总线都是 `can0`。

### 4.2 can1：PP 电机

can1 上当前包含 5 台 PP 电机：

- `elbow_pitch_joint` -> `0x1F`，机械臂第 3 轴
- `wrist_pitch_joint` -> `0x20`，机械臂第 4 轴
- `wrist_roll_joint` -> `0x21`，机械臂第 5 轴
- `wrist_yaw_joint` -> `0x22`，机械臂第 6 轴
- `left_gripper_finger_joint` -> `0x1E`，夹爪主动轴

说明：

- can1 电机统一通过位置控制参与 ros_control。
- 当前正式测试路径不走独立 `canopen_hw_node`，也不单独起历史 CANopen bringup。

### 4.3 ECB 网络：摆臂电机

ECB 网络当前包含 4 台摆臂电机：

- `ecb_joint_02`
- `ecb_joint_03`
- `ecb_joint_04`
- `ecb_joint_05`

说明：

- 这 4 台电机不占用 `can0/can1`。
- 它们走 `ecb://192.168.1.30`，由 `can_driver` 统一托管。
- 摆臂测试统一采用位置模式，不再使用速度点动作为标准验收路径。

### 4.4 分类联调 launch 与生命周期

每次只测一类时，不要直接用全量 `can_driver.yaml` 起底层节点，否则生命周期会把未接入的其它总线也纳入健康检查。按分类使用下面三个入口，它们会只加载本类 joints，并自动执行 `init -> enable -> resume`：

```bash
# can0: 前轮 MT + 机械臂前 2 轴 MT
roslaunch can_driver can_driver_can0_test.launch

# can1: 机械臂后 4 轴 PP + 夹爪主动轴
roslaunch can_driver can_driver_can1_test.launch

# ECB 网络: 摆臂 4 轴
roslaunch can_driver can_driver_ecb_network_test.launch
```

启动后在另一个终端确认生命周期：

```bash
rostopic echo -n 1 /can_driver_node/lifecycle_state
```

期望 `data: "Running"`。若不是 Running，不要继续发运动命令，先看当前分类 launch 终端里的 init/enable/resume 日志。

若日志中出现类似：

```text
Startup feedback sync timed out on device 'can0'
tx_ok=16 ... rx_ok=0
```

含义是驱动已经向 `can0` 发出查询帧，但没有收到任何电机反馈。此时不是 launch 没起来，也不是生命周期服务缺失，而是当前分类总线没有回包。先按下面顺序排查：

```bash
ip -details -statistics link show can0
python3 /home/fire/catkin_ws/src/robot24/can_driver/scripts/socketcan_scan_ids.py \
  --device can0 \
  --protocol mt \
  --min-id 1 \
  --max-id 31 \
  --show-can-id-hex \
  --debug-tx
```

- 如果 scan 也只有发送没有回复，检查电机电源、CANH/CANL、终端电阻、波特率、`can0`/`can1` 是否接反。
- 如果 scan 能发现在线 ID，但 lifecycle 仍失败，优先检查 overlay 里的 `motor_id` 是否和扫描结果一致。
- 如果只接了部分 can0 电机，不要用包含 4 台 can0 电机的完整分类入口，应先改成只加载已接入电机的 overlay 后再 init；分类 launch 支持通过 `joints_overlay:=...` 显式指定现场临时 overlay。

注意：三个分类 launch 都是低层直通测试入口，命名空间为 `/can_driver_node`，适合 `mt_motor_interface.py`、`test_ecb_*.sh`、`demo_motor_move.sh` 这类脚本。整车回归、MoveIt、底盘上层、摆臂上层 UI 仍应回到 `hybrid_motor_hw.launch` 或 `full_system.launch`。

## 5. can0 调试方案：底盘前轮 + 机械臂前两轴 MT

### 5.1 目标

验证 MT 电机能完成：

- 使能 / 失能
- 底盘 `0x141`、`0x142` 的速度模式移动
- 机械臂前两轴 `0x143`、`0x144` 的位置模式基础动作

### 5.2 配置检查

- `can_driver/config/can_driver.yaml` 中 MT 关节应为 `protocol: MT`。
- `can_driver/config/mt_control_profiles.yaml` 中 profile 和 `mt_motor_ids` 正确。

若要直接扫描 can0 上的 MT 电机，可使用同目录下的 SocketCAN 扫描脚本。
注意 MT 协议在总线上探测的是节点号（node_id），而现场配置里常写的是完整控制帧 ID（motor_id）。
两者关系为：`motor_id = 0x140 + node_id`。
脚本会先打印当前配置中的 MT ID，再扫描 can0 上真实在线的 MT 节点，用于发现配置是否写错。
若扫描结果与配置不一致，应以扫描出的在线结果为准，再回填配置。

当前仓库配置里机械臂前两轴暂写为：

- `shoulder_yaw_joint`: `node_id=0x03` -> `motor_id=0x143`
- `shoulder_pitch_joint`: `node_id=0x04` -> `motor_id=0x144`

扫描命令：

```bash
python3 /home/fire/catkin_ws/src/robot24/can_driver/scripts/socketcan_scan_ids.py \
  --device can0 \
  --protocol arm-mt \
  --show-can-id-hex
```

期望输出类似：

```text
[scan] device=can0 protocol=arm-mt range=1-31
```

## 9. 全链路联调顺序（推荐）

1. 仅起统一门面 + 生命周期（不跑 MoveIt）
2. can0 单轴：`0x141`、`0x142`、`0x143`、`0x144`
3. can0 底盘 `/cmd_vel` 链路
4. can1 单轴：`0x1F`、`0x20`、`0x21`、`0x22`、`0x1E`
5. ECB 单轴 -> ECB 分组
6. flipper_control 位置模式轨迹
7. 六轴统一 Action 轨迹
8. 夹爪单电机开合
7. 最后再起 full_system 做整机回归

整机启动命令：

```bash
roslaunch robot_bringup full_system.launch \
  enable_moveit:=true \
  enable_mobility_control:=true \
  enable_flipper_control:=true \
  enable_gripper_cmd:=true
```

当前整机正式入口默认会自动推进 unified lifecycle 到 `Running`。
若只做总车中的单总线回归，也可直接透传分模块 profile，例如：

```bash
roslaunch robot_bringup full_system.launch \
  can_interface_profile:=can1 \
  enable_moveit:=false \
  enable_mobility_control:=false \
  enable_flipper_control:=false \
  enable_gripper_cmd:=true
```

若现场需要分阶段排障或保留人工 lifecycle 操作，需显式覆盖：

```bash
roslaunch robot_bringup full_system.launch \
  profile_auto_lifecycle:=false \
  auto_init:=false auto_enable:=false auto_release:=false
```

整机回归前再做一次快速确认：

```bash
rosnode list | egrep 'canopen_hw_node|hybrid_motor_hw_node|flipper_control|move_group'
```

期望是：

- 使用 `full_system.launch` 时应有 `/hybrid_motor_hw_node`
- 不应同时出现独立历史 `/canopen_hw_node`
- `flipper_control` 应接在 hybrid 后端，而不是独立 CANopen 后端

## 10. 常见故障与处理

1. 生命周期卡在 `Faulted`

- 先看 `motor_states` 哪个轴 fault。
- 执行：

```bash
rosservice call /hybrid_motor_hw_node/recover "{}"
rosservice call /hybrid_motor_hw_node/enable "{}"
rosservice call /hybrid_motor_hw_node/resume "{}"
```

2. ECB 发现不到设备

- 检查网口是否真实 `UP` 且在 `192.168.1.x`。
- 执行 logtool 权限脚本并 `newgrp pcap`。

3. 机械臂可规划不可执行

- `arm_position_controller` 未 running，或 joint 集合不一致。
- 检查：

```bash
rosservice call /controller_manager/list_controllers
rostopic echo -n 1 /joint_states
```

4. 夹爪无动作

- 确认 `gripper_controller` 已启动。
- 确认只控 `left_gripper_finger_joint`，右指不应独立下发。
