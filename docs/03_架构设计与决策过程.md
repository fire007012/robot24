# 架构设计与决策过程

## 1. 核心需求

- MoveIt 发送 6 轴轨迹控制机械臂，不感知底层驱动差异
- Navigation 通过 `/cmd_vel` 驱动底盘
- 摆臂对外暴露位置接口（当前遥控，未来自主翻越）
- **两个驱动包独立运行，不修改现有代码**

## 2. 关键架构约束

### 2.1 一个 Controller 不能跨 controller_manager

每个驱动节点有自己独立的 `controller_manager`。`JointTrajectoryController`
在加载时向所在 `controller_manager` 的 `RobotHW` 请求 joint handle。

如果配了 6 个 joint 但 RobotHW 里只注册了 4 个，controller 会拒绝加载。

**因此不可能在任何一个节点里创建一个管全部 6 轴臂关节的 arm_controller。**

### 2.2 MoveIt 只认一个 Action Server

MoveIt 需要向一个 `FollowJointTrajectory` action server 发送包含全部 6 轴的轨迹。
它不会帮你拆分轨迹。

### 2.3 一条 CAN 线 = 一个驱动节点

多个进程打开同一条 CAN 总线会导致：
- 写入时序竞争
- DeviceManager 内部协议状态 / 刷新线程 / mutex 冲突
- 收帧时需互相过滤，增加复杂度

故障域与物理域一致，一条线一个节点是最自然的边界。

## 3. 候选方案评估

### 方案 A：按部署切换（一次只用一个包）

不满足需求。系统中 6 轴机械臂的关节分属两个协议，必须同时运行两个包。

### 方案 B：合并 RobotHW（写一个 UnifiedRobotHW）

在一个 RobotHW 内同时持有两个协议后端。
需要把 ros_canopen 当库用，理解 `canopen_master` / `canopen_402` 的 API。
代码耦合度高，工作量中等偏大。

### 方案 C：combined_robot_hw

ROS 标准机制，用 pluginlib 把两个 RobotHW 组合。
但 `canopen_motor_node` 不是标准 RobotHW 插件（是独立节点），
需要额外封装 CanopenRobotHW 插件。同时存在两个 socket 打开同一条 CAN 线的问题。

### 方案 D：Action 代理分发器 ✅ 最终选定

两个驱动节点完全独立运行，各自有自己的 controller_manager 和 JointTrajectoryController。
中间加一个 C++ 代理节点：
- 收到 MoveIt 的 6 轴轨迹
- 按关节名拆成 4 + 2 两组
- 同时发给两个 action server
- 等两边完成后返回合并结果

**选定理由：**

| 优势 | 说明 |
|------|------|
| 不改现有代码 | 两个驱动包原样保留 |
| 实现简单 | 代理节点 ~300-400 行 C++ |
| 无总线冲突 | 两条 CAN 线本来就独立 |
| 延迟可忽略 | 轨迹是一次性传输，不是实时流 |
| 精度可接受 | 两个后端开始时刻差 ~1-2ms，实验室场景无影响 |

## 4. 关于代理节点延迟的分析

### FollowJointTrajectory 不是实时控制流

MoveIt 发出的是一整条轨迹计划（一次性消息），包含带时间戳的路点序列。
代理节点只做一次拆分和转发，额外延迟 ~0.5-1ms。

### 实时控制发生在节点内部

真正的实时插值和位置下发发生在每个节点内部的 100Hz 控制循环：

read() → JointTrajectoryController 按时间戳插值 → write() → CAN 帧下发

这个循环完全不经过代理节点。代理节点在轨迹下发后只是等待结果。

### 延迟来源分析

| 环节 | 延迟 | 影响 |
|------|------|------|
| MoveIt → 代理节点 | ~0.5ms | 可忽略，只传一次 |
| 代理节点拆分 + 转发 | ~0.1ms | 可忽略 |
| 两个后端收到轨迹的时间差 | ~0.5-2ms | 唯一真实偏差源 |
| 各自 100Hz 控制环路 | 每周期 10ms | 与代理无关 |

在 3 秒的运动周期里，1-2ms 的偏差为 0.05%，对实验室场景完全可接受。

---

## 5. 关于 can_driver_node 职责是否膨胀的讨论

can_driver_node 同时管臂关节和底盘履带，看起来职责多，但实际上：

- 它的角色和 `canopen_motor_node` 完全对称——都是"一条 CAN 线的纯驱动"
- `JointTrajectoryController`、`diff_drive_controller` 是独立的 controller 插件
- 这些插件靠配置加载进 controller_manager，不是 can_driver_node 的代码
- 故障域与物理域一致：一条线的电机同生共死

强行拆分反而会引入新问题：多进程共享同一条 CAN 线、DeviceManager 状态冲突。
