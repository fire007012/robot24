# 电机生命周期管理约定（2026-04-15）

## 1. 文档目的

本文用于明确当前工程中“电机初始化、使能、停机、恢复”等生命周期能力到底归谁管理，避免在上层控制包、上位机、bringup 脚本和底层执行节点之间继续重复建设相同职责。

本文结论覆盖当前工作区的现行设计。

---

## 2. 当前结论

当前正式结论如下：

- 不新增独立的“电机生命周期控制包”
- 电机生命周期 authority 统一收口到 `Eyou_ROS1_Master/hybrid_motor_hw_node`
- 上位机/HMI 可以作为生命周期操作发起方，但不是生命周期状态机 owner
- `arm_control`、`mobility_control`、`flipper_control` 等上层功能包不再各自维护一套生命周期逻辑
- 若后续多个上层模块都需要复用相同调用流程，只补客户端辅助库，不新增新的生命周期 ROS 节点

一句话概括：

`Eyou_ROS1_Master` 负责“真正执行和维护生命周期状态机”，上位机或人工操作负责“决定什么时候触发这些动作”。

---

## 3. 正式生命周期 authority

当前正式 authority 为：

- launch：`Eyou_ROS1_Master/launch/hybrid_motor_hw.launch`
- node：`Eyou_ROS1_Master/hybrid_motor_hw_node`

该节点已经承担以下职责：

- 组合 `can_driver` 与 `Eyou_Canopen_Master`
- 对外暴露统一生命周期服务
- 在双后端之间执行统一状态迁移
- 在后端生命周期请求失败时进行回滚或失败收敛
- 托管统一 `controller_manager`

因此，生命周期 authority 已经存在，当前不再需要在其上方重复封装一个新的“生命周期节点”。

---

## 4. 为什么不再单独新建生命周期包

不单独新建生命周期包，主要因为当前若新增一层，其南向接口与北向接口将几乎同形：

- 南向仍然是 `init/enable/disable/halt/resume/recover/shutdown`
- 北向大概率还是 `init/enable/disable/halt/resume/recover/shutdown`

这类包通常没有形成新的抽象，只是在现有 facade 之上再包一层 facade。

当前方案下，如果新增单独生命周期包，会带来以下问题：

- 生命周期概念重复出现，职责边界继续模糊
- 状态真相源不唯一，容易出现“上层认为已初始化、底层实际未收敛”的问题
- 故障恢复链被人为拉长，不利于排查
- 上层控制包可能开始错误承担底层执行职责

因此当前设计原则是：

- 生命周期状态机只保留一份
- 生命周期服务入口只保留一份正式入口
- 上层只做调用，不再做 authority

---

## 5. 当前对外生命周期接口

`hybrid_motor_hw_node` 当前正式对外提供以下生命周期服务：

- `init`
- `enable`
- `disable`
- `halt`
- `resume`
- `recover`
- `shutdown`

这些服务已经是当前系统的正式入口。

在语义上可理解为：

| 服务 | 作用 |
|------|------|
| `init` | 建立后端初始化链路，使系统进入可上电准备态 |
| `enable` | 从待机态推进到 Armed |
| `resume` | 从 Armed 进入 Running |
| `halt` | 从 Running 回到 Armed，冻结运动输出 |
| `disable` | 从 Running/Armed 回到 Standby |
| `recover` | 从 Faulted 尝试恢复到可继续操作的状态 |
| `shutdown` | 关闭当前执行链，回到更低生命周期状态 |

说明：

- 这些语义以 `Eyou_ROS1_Master` 当前实现为准
- 上层包不应重新定义同名但不同语义的服务

---

## 6. 上位机、bringup、上层控制包的职责分工

### 6.1 `Eyou_ROS1_Master`

负责：

- 生命周期状态机
- 生命周期服务入口
- 底层后端一致性
- 回滚与失败收敛
- 电机执行主链

不负责：

- 操作员交互界面
- 任务级调度
- 业务级“什么时候允许开始作业”的决策

### 6.2 上位机 / HMI

负责：

- 提供人工按钮或任务入口
- 根据作业流程调用生命周期服务
- 显示当前系统状态
- 在故障或未就绪时阻止用户继续下发运动目标

不负责：

- 维护生命周期真相状态
- 实现新的底层状态机
- 绕开 `Eyou_ROS1_Master` 直接操作底层后端

换句话说：

- 上位机可以“发起初始化”
- 但不能“拥有初始化状态机”

### 6.3 `robot_bringup`

负责：

- 启动节点
- 传递启动参数
- 决定是否启用自动启动参数

不负责：

- 替代生命周期 service 做业务逻辑判断
- 作为长期运行时的生命周期 authority

### 6.4 `arm_control` / `mobility_control` / `flipper_control`

负责：

- 目标表达
- 模式管理
- 安全约束
- 在运行前检查生命周期是否允许动作

不负责：

- 重新实现 `init/enable/disable/...`
- 自己维护一套电机状态机
- 直接接管底层电机执行

## 7. 当前推荐使用方式

### 7.1 实机默认策略

当前更推荐的实机策略为：

- `auto_init=false`
- `auto_enable=false`
- `auto_release=false`

原因：

- 启动后先保持人为确认
- 避免上电即自动进入运动相关状态
- 便于实机联调和故障定位

### 7.2 上位机最小按钮集

上位机或调试终端建议提供以下最小操作按钮：

1. 初始化
2. 进入运行
3. 停到待机
4. 故障恢复
5. 关闭执行链

建议将它们对应到以下服务组合：

| 上位机动作 | 推荐服务链 |
|------------|------------|
| 初始化 | `init` |
| 进入运行 | `enable -> resume` |
| 停到待机 | `halt -> disable` 或直接 `disable` |
| 故障恢复 | `recover`，必要时再 `enable -> resume` |
| 关闭执行链 | `shutdown` |

### 7.3 上层控制包的运行前检查

上层控制包在发送运动命令前应至少满足以下条件：

- 生命周期已处于允许运动的状态
- 目标关节无 fault
- 目标关节在线

若不满足，则应：

- 拒绝发送运动目标
- 给出清晰错误信息
- 引导用户先处理生命周期问题

## 8. 当前建议的工程约定

为避免后续继续长出重复层，当前建议遵守以下约定：

### 8.1 只保留一个正式生命周期入口

正式入口固定为：

- `Eyou_ROS1_Master/hybrid_motor_hw_node`

后续文档、launch、上层包说明中，若提到“电机初始化/使能/恢复”，均默认指向该节点。

### 8.2 不新增生命周期控制节点

除非后续真的出现以下新增职责，否则不应新增独立生命周期节点：

- 多来源生命周期仲裁
- 电源、急停、控制器切换的统一编排
- 运行权限审批或互锁控制
- 跨机器人或跨控制柜的统一调度

在当前工程阶段，上述需求都还没有成立。

### 8.3 如需复用调用逻辑，只做客户端辅助库

如果后续多个包都需要重复写同样的 service client 流程，建议只补：

- Python 客户端辅助模块
- 或 C++ 客户端辅助类

而不是新起一个 ROS 节点去转发同样的服务。

---

## 9. 当前状态反馈的使用建议

当前 `Eyou_ROS1_Master` 已经提供按关节的运行状态输出。

因此上层建议按以下方式消费状态：

- 生命周期总入口状态：以 `hybrid_motor_hw_node` 当前服务调用结果和运行态为准
- 关节在线/故障细节：以 `joint_runtime_states` 为准

若后续确实需要单独的“全局生命周期状态 topic”，应优先在 `Eyou_ROS1_Master` 内补齐，而不是在外部再套一个新的生命周期包去拼装状态。

---

## 10. 与当前控制包拆分方案的关系

本约定与当前控制包拆分方案保持一致：

- `Eyou_ROS1_Master`：正式统一执行入口
- `arm_control`：机械臂上层控制
- `mobility_control`：底盘上层控制
- `flipper_control`：四摆臂上层控制
- `car_control`：迁移期保留

也就是说，后续包结构中：

- 不新增 `motor_lifecycle_control`
- 不新增单独“电机执行节点”
- 不新增新的生命周期 authority

---

## 11. 后续落地建议

当前建议的后续动作顺序为：

1. 统一文档口径，明确生命周期 authority 在 `Eyou_ROS1_Master`
2. 上位机界面直接对接 `hybrid_motor_hw_node` 的生命周期服务
3. 上层控制包只保留“运行前检查”和“错误提示”
4. 若后续发现 service client 逻辑重复，再抽共享客户端工具

不建议的顺序为：

1. 先新建一个生命周期包
2. 再在其上重复暴露同名服务
3. 最后再回头解释它和 `Eyou_ROS1_Master` 的关系

这会再次把职责边界做模糊。

---

## 12. 最终结论

当前工程中：

- 电机生命周期由 `Eyou_ROS1_Master/hybrid_motor_hw_node` 管理
- 上位机负责发起，不负责拥有生命周期状态机
- 上层控制包负责配合，不负责重建生命周期体系

因此，当前阶段的正确方向不是“继续新增生命周期包”，而是“把现有正式入口用清楚、用统一”。
