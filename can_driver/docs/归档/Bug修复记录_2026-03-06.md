## 关键实现说明

1. `0xA4` 速度单位修正
- 协议：`DATA[2..3] = uint16_t maxSpeed (1 dps/LSB)`
- 现实现：`commandedVelocity` 仍沿用 `0xA2` 语义（`0.01 dps/LSB`），发送 `0xA4` 前做四舍五入换算：
  - `absVelDps = (absVel001Dps + 50) / 100`
  - 非零输入最小下限保护为 `1 dps`
  - 零输入沿用默认 `100 dps`

2. 速度缓存语义统一
- `state.velocity` 从“RPM（内部除以6）”改为“协议原始 dps”
- `getVelocity()` 语义随之变更：返回 `1 dps/LSB`
- 头文件注释已同步，避免后续调用方误解

3. `0x9A` 电压/保留字段留存
- 协议中 `DATA[2..5]` 不同机型语义可能不同（电压或保留）
- 本次先做原始值缓存，避免信息丢失，后续再按具体电机型号定义工程单位

## 兼容性与影响评估

1. 破坏性变更：`MtCan::getVelocity()` 单位由 RPM 改为 dps
- 如果上层 `velocityScale` 之前按 RPM 配置，则需要调整为按 dps 换算。
- 常见换算参考：
  - RPM -> rad/s：`2*pi/60`
  - dps -> rad/s：`pi/180`

2. 非破坏性变更
- `0xA4` 单位修正仅影响此前错误速度的场景，属于安全性修复。
- `0x9A` 增加解析字段为增量能力，不影响现有接口行为。
- `0xA9` 回复补齐仅提升兼容性，不改变已有命令发送逻辑。

## 暂未在本次修改中处理的项

1. `setPosition(int32_t)` 与 `getPosition(int64_t)` 位宽不对称  
原因：受 `CanProtocol` 统一接口约束，调整会涉及跨协议接口变更。

2. `0xA4 spinDirection` 仍固定 `0x00`  
原因：当前上层未暴露方向策略参数，需评估 API 扩展方案后再改。

3. `iq` 字段物理单位待协议机型确认  
本次保持原逻辑：`rawCurrent / 100.0`。
