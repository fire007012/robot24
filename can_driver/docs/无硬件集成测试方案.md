# can_driver 无硬件集成测试方案

日期：2026-03-08
范围：`can_driver` 包（ROS1 + SocketCAN + MT/PP 协议）

## 1. 目标与约束

目标：在不接入真实 CAN 设备和电机的前提下，验证 `can_driver_node` 的端到端行为，覆盖初始化、控制循环、服务接口、状态发布、故障与并发路径。

约束：
- CI 环境可能无 root 权限，不能依赖创建 `vcan`。
- 测试应可在 `catkin_make run_tests_can_driver` 下稳定执行。
- 保留一组可选 `vcan` 测试用于“接近真实 SocketCAN 栈”的回归验证。

## 2. 现状与缺口

当前已有：
- Level-1/2 单元测试覆盖较好：`SafeCommand`、`JointConfigParser`、`MtCan/EyouCan`、`SocketCanController`、`DeviceManager`。
- 现有测试文件位于 `tests/`，多数是协议与工具类级别。

主要缺口：
- 缺少 `CanDriverHW` 级别的“节点行为集成测试”（服务 + read/write + topic + 限位 + direct 命令超时）。
- 缺少无硬件故障注入与并发压力验证。

## 3. 总体方案（3 层）

### 3.1 L3-A：纯软件集成（主线，CI 必跑）

思路：
- 在测试中注入 `FakeDeviceManager/FakeProtocol/FakeTransport`，不走真实 socketcan。
- 启动 `ros::init` + `CanDriverHW` 实例，驱动 `read()/write()`，调用 `~init/~shutdown/~recover/~motor_command`，订阅 `~motor_states`。

优点：
- 不依赖系统网络设备与权限，稳定、可重复。
- 可精确故障注入（设备未就绪、协议拒绝、抛异常）。
- 执行快，适合 PR 级别门禁。

### 3.2 L3-B：vcan 近实物集成（可选，Nightly/手动）

思路：
- 使用 `vcan0/vcan1 + loopback=true`，验证真实 `SocketCanController` 与协议层互动。
- 无 vcan 时自动 `GTEST_SKIP()`（与现有 `test_device_manager.cpp` 风格一致）。

优点：
- 补齐部分真实 CAN 栈路径。

限制：
- 环境依赖强，不能作为唯一门禁。

### 3.3 L3-C：并发与鲁棒性（CI 可选必跑，TSAN Nightly）

思路：
- 多线程并发执行 `read/write/service/topic`，验证无死锁、无竞态、状态一致。
- TSAN 构建单独运行，观察 data race 报告。

## 4. 测试可观测面

基于当前接口，统一从以下可观测面断言：
- Service 响应：`success/message` 是否符合预期。
- 协议调用痕迹：`setPosition/setVelocity/Enable/Disable/Stop/setMode` 调用参数与次数。
- 发布消息：`~motor_states` 内容（`motor_id/name/position/velocity/current/enabled/fault/mode`）。
- 命令选择逻辑：direct topic vs ros_control 命令回退。
- 限位/缩放：发送值是否先钳制再 scale，再转 `int32`。

## 5. 用例矩阵（首批必须覆盖）

### 5.1 生命周期与服务

1. `init` 成功：
- 前置：合法 joints 参数，Fake transport ready。
- 断言：`init.success=true`，设备被初始化，`active_` 生效，`motor_states` 开始发布。

2. `init` 失败：
- 前置：缺少 `joints` 或非法 protocol/control_mode。
- 断言：`CanDriverHW::init()` 返回 false。

3. `shutdown`：
- 断言：`active_=false` 后不再下发命令；direct 命令缓存清空。

4. `recover`：
- 覆盖 `motor_id=具体值 / 0xFFFF / 0(兼容语义)`。
- 覆盖失败分支：device not ready / protocol unavailable / rejected / exception。

5. `motor_command`：
- 覆盖 `ENABLE/DISABLE/STOP/SET_MODE/unknown`。
- `SET_MODE` 覆盖非法 value（非 0/1）。

### 5.2 read/write 控制循环

1. `read()`：
- 协议缓存原始值 -> 按 scale 变换到 `joint pos/vel/eff`。

2. `write()` velocity/position：
- `ros_control` 命令经限位与 scale 后下发。
- 设备 not ready 时跳过并保持健壮。

3. direct 命令优先级与超时：
- 超时前使用 direct。
- 超时后自动回退 ros_control。
- `debug_bypass_ros_control=true` 时，未收到 direct 不下发 fallback。

4. 限位覆盖：
- URDF 限位生效。
- rosparam 覆盖 URDF。
- 无穷/NaN 与极值输入不导致崩溃。

### 5.3 topic 发布语义

1. `motor_states` 周期发布：
- 多 joint 均能收到。

2. 字段语义：
- `position/velocity/current` 为“反 scale 后并钳制到消息整数范围”的结果。
- `enabled/fault` 与协议缓存一致。

### 5.4 多设备/多协议隔离

1. `vcan0 + MT` 与 `vcan1 + PP` 同时存在：
- 命令分发到正确协议实例，互不串扰。

2. 同设备多协议：
- `MT/PP` group 都可写入，设备级互斥不死锁。

## 6. 建议的最小可测试性改造

为了实现 L3-A（纯软件集成），建议做最小注入改造：

1. 抽象设备管理接口
- 新增 `IDeviceManager`（接口与 `DeviceManager` 对齐）。
- `CanDriverHW` 默认使用真实 `DeviceManager`；测试传入 fake。

2. 暴露测试构造路径
- 新增构造：`CanDriverHW(std::shared_ptr<IDeviceManager> dm)`。
- 生产代码路径不变，兼容当前 launch。

3. 保持外部行为不变
- 不改 service/topic 名称与参数语义。
- 不把测试开关混入运行参数。

这样可在不依赖 vcan 的情况下完成完整集成断言。

## 7. 目录与文件建议

新增：
- `tests/fakes/FakeTransport.h`
- `tests/fakes/FakeProtocol.h`
- `tests/fakes/FakeDeviceManager.h`
- `tests/test_can_driver_hw_integration.cpp`
- `tests/test_can_driver_hw_services.cpp`

可选新增（vcan）：
- `tests/test_can_driver_hw_vcan.cpp`

CMake：
- 新增以上 gtest 目标并链接 `can_driver_transport`。
- `test_can_driver_hw_vcan.cpp` 在运行时探测 `/sys/class/net/vcan0`，无设备则 `GTEST_SKIP()`。

## 8. CI 分层执行

PR 必跑：
- 现有单元测试 + L3-A（Fake 集成）

Nightly/手动：
- L3-B（vcan）
- L3-C（TSAN 压测）

推荐命令：
- `catkin_make --pkg can_driver -DCMAKE_BUILD_TYPE=RelWithDebInfo`
- `catkin_make run_tests_can_driver`
- `catkin_test_results`

TSAN（单独构建目录）：
- `-DCMAKE_CXX_FLAGS="-fsanitize=thread -fno-omit-frame-pointer"`
- `-DCMAKE_EXE_LINKER_FLAGS="-fsanitize=thread"`

## 9. 验收标准（无硬件）

达到以下条件即认为“无硬件集成测试体系可用”：
- `CanDriverHW` 关键路径（init/read/write/services/motor_states/direct timeout）均有自动化用例。
- PR 流水线稳定通过，失败可复现。
- 至少 1 组并发压力测试稳定通过（无死锁/崩溃）。
- vcan 测试作为增强项，不影响主线 CI 可用性。

## 10. 任务拆分（建议两周）

第一阶段（2-3 天）：
- 完成 `IDeviceManager` 注入改造与回归。
- 加 2 个冒烟用例：`init + write`、`motor_command`。

第二阶段（3-4 天）：
- 补齐 direct timeout、限位、motor_states 语义用例。
- 接入 PR CI。

第三阶段（2-3 天）：
- 并发压力测试 + TSAN Nightly。
- 可选 vcan 集成测试。
