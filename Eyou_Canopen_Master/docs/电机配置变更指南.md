# Eyou_Canopen_Master 电机配置变更指南

本文档面向“我要改 CANopen 电机配置”这个场景，重点说明：

- 应该改 `joints.yaml` 还是 `master.yaml`
- 哪些值必须两边同步
- 改完以后为什么一定要重新生成 `master.dcf`
- 增删电机时还要同步哪些控制器文件

适用文件：

- `config/joints.yaml`
- `config/master.yaml`
- `config/master.dcf`
- `config/ros/controllers.yaml`
- `config/joints_flipper_4axis.yaml`
- `config/master_flipper_4axis.yaml`
- `config/master_flipper_4axis.dcf`
- `config/ros/controllers_flipper_4axis.yaml`

## 1. 先分清两类配置

`Eyou_Canopen_Master` 里和电机有关的配置分成两层：

| 文件 | 作用 | 什么时候必须改 |
| --- | --- | --- |
| `joints.yaml` / `joints_flipper_4axis.yaml` | 运行时配置。定义 joint 名、node_id、默认模式、换算系数、软限位相关参数、轨迹约束。 | 改 joint 名、node_id、默认模式、换算参数、限位相关参数时。 |
| `master.yaml` / `master_flipper_4axis.yaml` | DCF 生成输入。定义主站、从站、PDO、TPDO、SDO、heartbeat、SYNC 等对象字典配置。 | 改 node_id、PDO 映射、SDO 启动写值、heartbeat、SYNC 周期、增删从站时。 |

运行时真正启动用的是 `master.dcf`，不是 `master.yaml` 本身。  
所以只改 `master.yaml` 而不重新生成 `master.dcf`，启动时不会按你的新配置工作。

## 2. 先确定你当前用的是哪套 profile

标准机械臂 profile：

- `config/joints.yaml`
- `config/master.yaml`
- `config/master.dcf`
- `config/ros/controllers.yaml`
- `launch/bringup.launch`

四摆臂 profile：

- `config/joints_flipper_4axis.yaml`
- `config/master_flipper_4axis.yaml`
- `config/master_flipper_4axis.dcf`
- `config/ros/controllers_flipper_4axis.yaml`
- `launch/bringup_flipper_4axis.launch`

不要混着改。现场启动哪个 launch，就改它对应的那套文件。

## 3. 改动项和必改文件的对应关系

| 变更内容 | 必改文件 | 还要做什么 |
| --- | --- | --- |
| 新增一个轴 | `joints*.yaml`、`master*.yaml` | 重新生成 `master*.dcf`，同步 `controllers*.yaml`，同步 URDF/上层包 |
| 删除或停用一个轴 | `joints*.yaml`、`master*.yaml` | 重新生成 `master*.dcf`，同步 `controllers*.yaml` |
| 修改 `node_id` | `joints*.yaml`、`master*.yaml` | 重新生成 `master*.dcf` |
| 修改 `default_mode` | `joints*.yaml` | 若启动 SDO 里写了 `0x6060`，同步改 `master*.yaml` 的 `sdo` |
| 修改 `verify_pdo_mapping` | `joints*.yaml` | 一般不用改 `master*.yaml` |
| 修改 `counts_per_rev` / `rated_torque_nm` / `velocity_scale` / `torque_scale` | `joints*.yaml` | 一般不用改 `master*.yaml` |
| 修改 PDO / TPDO / heartbeat / SYNC / 启动 SDO | `master*.yaml` | 重新生成 `master*.dcf` |
| 修改控制器接管哪些 joint | `controllers*.yaml` | 必须保证 joint 名与 `joints*.yaml` 一致 |

## 4. `joints.yaml` 这一层怎么写

一个最小合法条目长这样：

```yaml
joints:
  - name: shoulder_yaw_joint
    canopen:
      node_id: 6
      verify_pdo_mapping: false
      default_mode: 7
    counts_per_rev: 6619136
    rated_torque_nm: 1.0
    max_velocity_for_clamp: 500000
```

### 4.1 代码硬校验

下面这些规则是 `LoadJointsYaml()` 直接校验的：

| 字段 | 规则 |
| --- | --- |
| `joints` | 必须存在，且必须是非空数组 |
| `canopen.node_id` / 顶层 `node_id` | 若填写，必须在 `1..127` |
| `canopen.default_mode` / 顶层 `default_mode` | 只能是 `7`、`8`、`9`、`10` |
| `ip_interpolation_period_ms` | 若填写，必须在 `1..255` |
| `counts_per_rev` | 必须 `> 0` |
| `counts_per_meter` | 若填写，必须 `> 0` |
| `rated_torque_nm` | 必须 `> 0` |
| `max_velocity_for_clamp` | 必须 `> 0` |

### 4.2 虽然代码不拦，但现场必须遵守的约束

以下约束当前解析器不会全拦住，但你应该把它们当成硬规则：

- `name` 必须全局唯一
- `node_id` 必须全局唯一
- `joints` 数组顺序要稳定，不要随意重排
- `controllers*.yaml` 里出现的 joint 名必须都能在这里找到

原因：

- `robot_hw_ros` 会按 `joints` 顺序建立轴索引
- 控制器、诊断和日志都以这个顺序和名字为基准
- 同一个 `node_id` 重复配置会让总线寻址和 SDO/PDO 行为混乱

### 4.3 重要字段说明

| 字段 | 用途 | 变更建议 |
| --- | --- | --- |
| `name` | ROS joint 名 | 一旦改名，控制器、URDF、MoveIt、上层包都要同步 |
| `canopen.node_id` | 从站 node id | 生产环境建议显式填写，不要依赖“按顺序自动补 node_id” |
| `canopen.verify_pdo_mapping` | 启动时是否做 PDO 映射校验 | 调试阶段可关，定型后建议按需开启 |
| `canopen.default_mode` | 运行时默认模式 | `7=IP`、`8=CSP`、`9=CSV`、`10=CST` |
| `counts_per_rev` | 位置换算基准 | 改错会导致位置量纲整体错误 |
| `counts_per_meter` | 线性关节软限位换算基准 | 只对 `prismatic` 关节有意义 |
| `rated_torque_nm` | 力矩换算基准 | 改错会影响 effort 语义 |
| `velocity_scale` / `torque_scale` | 速度/力矩缩放 | 仅在确认设备语义后再改 |
| `ip_*` | IP 轨迹执行器参数 | 只在 IP 轨迹链路下生效 |

### 4.4 软限位相关的高风险点

如果 `canopen.auto_write_soft_limits_from_urdf: true`，那么每次 `init` 成功后都会尝试按 URDF 自动写软件限位。

这时要额外满足：

- `robot_description` 中必须存在同名 joint
- 该 joint 必须有完整的位置上下限
- 若 joint 是 `prismatic`，必须同时正确配置 `counts_per_meter`

否则问题不会拖到运行时，而是直接在 `init` 阶段失败。

### 4.5 `default_mode` 改了，不要只改一边

例如你把某轴从 CSP 改成 IP：

1. `joints.yaml` 里的 `canopen.default_mode` 要改
2. `master.yaml` 中该从站 `sdo` 里的 `0x6060` 启动写值也要改

否则会出现“运行时想按一种模式工作，主站启动时却先把驱动器切到另一种模式”的不一致。

## 5. `master.yaml` 这一层怎么写

`master.yaml` 决定的是主站下发到从站的对象字典配置，不是本地换算参数。

一个从站段通常长这样：

```yaml
shoulder_yaw_joint:
  node_id: 5
  dcf: /abs/path/YiyouServo_V1.4.dcfgen.eds
  heartbeat_producer: 500
  rpdo:
    1:
      cob_id: 0x205
      transmission: 1
      mapping:
        - {index: 0x6040, sub_index: 0, bits: 16}
        - {index: 0x607A, sub_index: 0, bits: 32}
        - {index: 0x6060, sub_index: 0, bits: 8}
  tpdo:
    1:
      cob_id: 0x185
      transmission: 1
      mapping:
        - {index: 0x6041, sub_index: 0, bits: 16}
        - {index: 0x6064, sub_index: 0, bits: 32}
        - {index: 0x6061, sub_index: 0, bits: 8}
  sdo:
    - {index: 0x6060, sub_index: 0, value: 7}
```

### 5.1 什么时候必须改 `master.yaml`

- 增删轴
- 改 `node_id`
- 改 PDO / TPDO 映射
- 改 heartbeat
- 改 `master.sync_period`
- 改启动时 SDO 写值

### 5.2 什么时候一般不用改 `master.yaml`

- 只改 `counts_per_rev`
- 只改 `rated_torque_nm`
- 只改 `velocity_scale` / `torque_scale`
- 只改 `verify_pdo_mapping`

这些属于运行时本地逻辑，通常只在 `joints.yaml` 生效。

## 6. 新增、修改、删除电机的标准流程

### 6.1 新增一个轴

1. 选定当前使用的 profile。
2. 在对应 `joints*.yaml` 中复制一段同类轴配置。
3. 显式填写新的 `name` 和 `canopen.node_id`。
4. 填好 `counts_per_rev`、`rated_torque_nm`、`default_mode` 等运行时参数。
5. 在对应 `master*.yaml` 中新增同名从站段，并保证 `node_id` 一致。
6. 如果默认模式改了，同步该段 `sdo` 中的 `0x6060` 写值。
7. 重新生成 DCF。
8. 把新 joint 加入对应 `controllers*.yaml`。
9. 同步 URDF 和上层控制包。

### 6.2 修改现有轴

推荐按下面顺序改，排障最轻松：

1. 先改 `joints*.yaml` 里的运行时参数
2. 若涉及 `node_id` / PDO / SDO，再改 `master*.yaml`
3. 重新生成 DCF
4. 再同步控制器 joint 列表

### 6.3 删除或停用一个轴

长期停用时，推荐直接从 profile 中删掉，而不是只让上层“不发命令”。

最小动作集：

1. 从 `joints*.yaml` 删除或注释该 joint
2. 从 `master*.yaml` 删除或注释对应从站段
3. 重新生成 `master*.dcf`
4. 从 `controllers*.yaml` 中删除该 joint
5. 若这是当前 launch 默认 profile，要确认 launch 指向的 controllers 文件仍然成立

四摆臂 profile 现在就是这样做的：2 轴机械臂段被整体注释掉，同时控制器文件里也只保留 4 个摆臂 joint。

## 7. DCF 重新生成命令

标准 profile：

```bash
source /opt/ros/noetic/setup.bash
CFG_DIR=$(rospack find Eyou_Canopen_Master)/config
cd "$CFG_DIR"

dcfgen -S -r -d . master.yaml
```

四摆臂 profile：

```bash
source /opt/ros/noetic/setup.bash
CFG_DIR=$(rospack find Eyou_Canopen_Master)/config
cd "$CFG_DIR"

dcfgen -S -r -d . master_flipper_4axis.yaml
cp master.dcf master_flipper_4axis.dcf
```

如果你改了 `master*.yaml` 却没重新生成 `master*.dcf`，启动时实际加载的仍是旧 DCF。

## 8. 控制器文件也必须同步

标准 profile 默认控制器文件是：

- `config/ros/controllers.yaml`

四摆臂 profile 默认控制器文件是：

- `config/ros/controllers_flipper_4axis.yaml`

必须保持：

- joint 名和 `joints*.yaml` 完全一致
- 控制器类型和默认模式相匹配

例子：

- `arm_position_controller` 适合位置语义链路
- `arm_velocity_controller` 适合速度语义链路
- `flipper_csp_controller` / `flipper_csv_controller` 分别对应四摆臂的 CSP / CSV 路径

## 9. 验证步骤

### 9.1 配置级验证

先确认 launch 指向的是你刚改过的文件：

- `bringup.launch` 默认读 `config/master.dcf`、`config/joints.yaml`、`config/ros/controllers.yaml`
- `bringup_flipper_4axis.launch` 默认读 4 轴那套文件

### 9.2 启动验证

```bash
roslaunch Eyou_Canopen_Master bringup.launch
```

或四摆臂：

```bash
roslaunch Eyou_Canopen_Master bringup_flipper_4axis.launch
```

然后检查：

- `rosservice list | grep canopen_hw_node`
- `rostopic echo /diagnostics`
- `rostopic echo /joint_states`

### 9.3 硬件联调验证

在硬件在线时继续做：

```bash
rosservice call /canopen_hw_node/init "{}"
```

如有需要，再做：

- `~/enable`
- `~/set_mode`
- `~/set_zero`
- `~/apply_limits`

如果 `verify_pdo_mapping=true` 且映射不一致，通常会在 `init` 或启动阶段暴露问题。

## 10. 最后检查清单

- 只改了当前真正使用的那套 profile
- `joints*.yaml` 与 `master*.yaml` 的 `node_id` 一致
- `default_mode` 与 `master*.yaml` 中 `0x6060` 启动写值一致
- 改过 `master*.yaml` 后已经重新生成 `master*.dcf`
- `controllers*.yaml` 的 joint 列表已经同步
- joint 名没有重复
- node_id 没有重复
- `counts_per_rev` / `rated_torque_nm` / `max_velocity_for_clamp` 都是正数
- 若启用了 `auto_write_soft_limits_from_urdf`，URDF 中对应 joint 的限位信息仍然完整

如果只记一条经验规则：  
`joints.yaml` 负责“本地怎么算、叫什么、默认按什么模式运行”；`master.yaml/master.dcf` 负责“主站开机到底往从站写什么对象字典配置”。两边不要只改一边。
