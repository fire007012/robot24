# CANopen 主站通信异常排查与修复报告（统一整理版）

**项目：** Eyou_Canopen_Master（Lely Core + ROS）  
**日期：** 2026-03-21  
**总线：** can0  
**从站：** Node ID = 5  
**控制模式：** CSP

---

## 1. 问题概述

本次现场排查中出现了两类故障：

1. 启动阶段：主站能发 SYNC/RPDO，但从站反馈（尤其 TPDO1）异常或缺失。
2. 系统阶段：控制器加载报错（`Could not find joint 'joint_2' in URDF model.`），并伴随节点关闭不稳定现象。

两份原始报告中存在部分重复与数值冲突（尤其身份对象字段），本报告已去重并统一为现场实测值和最终修复结果。

---

## 2. 关键现场现象

### 2.1 抓包现象（故障期）

- 常见帧：`0x080`（SYNC）、`0x205`（RPDO1）、`0x285`（TPDO2）
- 异常：`0x185`（TPDO1）出现频率低或阶段性缺失
- 历史问题：`0x305/0x405` 曾持续发送但从站侧映射为空，属于无效负载

### 2.2 控制器加载异常

- `arm_velocity_controller` 初始化失败，日志明确指向 URDF 缺失 `joint_2`。

---

## 3. 排查结论（统一版）

### 3.1 根因 A：EDS/DCF 生成前置错误

`dcfgen` 初始失败，阻断了稳定配置流程：

- `DynamicChannels` 段不被当前 `dcf-tools 2.3.5` 接受
- `[2023]`、`[2025]` 的 `UNSIGNED16` 上限写成 `65536`（溢出）

这导致配置文件生成与回归验证不稳定。

### 3.2 根因 B：身份对象不一致，导致 Boot 配置下载中止

Lely 在 boot 阶段会校验设备身份（1000h/1018h）后再执行配置下载。旧配置中身份字段不一致，导致“配置未按预期生效”的链路问题。

**统一后的现场实测身份值（已确认）：**

| 对象 | 含义 | 实测值 |
|---|---|---|
| `0x1000:00` | Device Type | `0x00220811` |
| `0x1018:01` | Vendor ID | `0x00000668` |
| `0x1018:02` | Product Code | `0x00221701` |

> 说明：原始草稿中出现过 `0x00000456/0x00000ABC/...` 等示例值，已统一为本次设备实测值，不再混用。

### 3.3 根因 C：URDF 关节定义与控制器配置不一致

`controllers.yaml` 期望 `joint_1~joint_6`，但原 `urdf/robot.urdf` 仅定义了 `joint_1`，导致控制器加载失败。

---

## 4. 已实施修复

### 4.1 EDS 修复（`config/YiyouServo_V1.4.dcfgen.eds`）

1. 删除 `[DynamicChannels]` 段（兼容 dcf-tools 2.3.5）
2. 修复越界：
   - `[2023] HighLimit: 65536 -> 65535`
   - `[2025] HighLimit: 65536 -> 65535`
3. 身份字段统一：
   - `[DeviceInfo] ProductNumber = 0x00221701`
   - `[1018sub2] DefaultValue = 0x00221701`

### 4.2 重新生成 DCF 产物

执行 `dcfgen master.yaml` 后，生成并提交：

- `config/master.dcf`
- `config/master.bin`
- `config/joint_1.bin`

### 4.3 URDF 修复

`urdf/robot.urdf` 从单关节扩展为 `joint_1~joint_6`，与 `config/ros/controllers.yaml` 对齐，修复 `joint_2` 缺失报错。

---

## 5. 验证结果

### 5.1 配置生成验证

- `dcfgen master.yaml` 通过（无 fatal error）

### 5.2 身份值验证（生成物）

`master.dcf` 中身份检查值已与现场实测一致：

- `1F84:5 = 0x00220811`
- `1F85:5 = 0x00000668`
- `1F86:5 = 0x00221701`

### 5.3 总线行为验证

- 已观测到 `0x185` 与 `0x285` 反馈恢复
- `0x305/0x405` 的无效发送问题已纳入配置修复路径（由 RPDO 映射与禁用策略控制）

---

## 6. 遗留与后续建议

1. **自动使能链路检查（建议下一步）**  
   当前状态机虽能计算 `controlword`，但需确认业务链路是否在周期中持续下发 `0x6040`，避免轴长期停留在非使能态。

2. **启动回归项固定化**  
   每次 EDS/固件变更后固定执行：
   - 身份对象（1000h/1018h）核对
   - `dcfgen` 生成
   - 启动后 SDO 下发抓包抽查

3. **发布流程建议**  
   对 `master.dcf/bin` 与 EDS 保持“同提交原子性”，避免现场出现文件版本漂移。
