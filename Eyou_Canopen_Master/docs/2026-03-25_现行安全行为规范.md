# 现行安全行为规范

日期：2026-03-25  
范围：`Eyou_Canopen_Master`

---

## 1. 适用性与优先级

本文档用于冻结当前代码基线下已经接受并回归验证的安全行为。

- 用途：作为联调、回归、后续重构的验收基线。
- 优先级：若本文档与 `docs/archive/2026-04-19_deprecated/0324import.md` 冲突，以本文档为准。
- 定位：`docs/archive/2026-04-19_deprecated/0324import.md` 保留为架构设计草案；本文档描述当前必须成立的行为契约。

---

## 2. 当前接受的系统语义

当前 `OperationalCoordinator` 采用如下对外语义：

- `RequestInit()`：`Configured -> Armed`
- `RequestEnable()`：`Standby -> Armed`
- `RequestRelease()`：`Armed -> Running`
- `RequestHalt()`：`Running -> Armed`
- `RequestRecover()`：`Faulted -> Standby`
- `RequestShutdown()`：`* -> Configured`

说明：

- `init -> Armed` 与 `recover -> Standby` 是当前接受的实现语义，不再要求与 `0324import` 原文逐字一致。
- 对安全性而言，关键不是 `init/recover` 的终态名称，而是进入 `Running` 的门控条件必须严格成立。

---

## 3. 必须保持的安全契约

### 3.1 Running 入口唯一性

- 只有 `RequestRelease()` 允许系统进入 `Running`。
- `Recover`、心跳恢复、fault 清除、`Init` 成功都不得直接让系统进入 `Running`。
- 任何故障、`Halt`、`Shutdown` 离开 `Running` 后，重新运动都必须再次经过显式 `Release`。

### 3.2 健康门控

- `RequestEnable()` 与 `RequestRelease()` 必须检查健康快照。
- 若任一轴存在 `is_fault=true`、`heartbeat_lost=true`，或 `global_fault=true`，则必须拒绝进入新的使能/运行状态。
- 系统不得允许“带病进入 Running”。

### 3.3 Recover 完成条件

- `RequestRecover()` 必须等待所有轴满足：
  - `is_fault == false`
  - `heartbeat_lost == false`
- 若在超时窗口内未满足条件，必须返回失败并保持 `Faulted`。
- `Recover` 成功仅表示故障已清除并回到可重新上电状态，不代表恢复运动。

### 3.4 使能前同步位置

这条是强制安全契约，不得回退。

- `RequestInit()`、`RequestRecover()`、`RequestShutdown()` 成功后，必须递增 `command_sync_sequence`。
- ROS 适配层在 `read()` 观察到以下任一事件时，必须立即触发预运动重同步：
  - `command_sync_sequence` 变化
  - `arm_epoch` 变化沿
  - `all_axes_halted_by_fault` 上升沿
- 触发重同步时，必须同时执行：
  - `pos_cmd = pos`
  - `valid = false`
  - guard 倒数重置
- guard 未结束前，不得把控制器旧命令标记为有效命令。
- 因此，系统在新使能会话开始前，必须先把位置命令对齐到当前位置，再允许控制器重新接管。

### 3.5 协议层解锁条件

- 协议层只有在以下条件同时满足时，才允许透传非冻结目标：
  - `cmd_valid == true`
  - `cmd_arm_epoch != 0`
  - `cmd_arm_epoch == arm_epoch`
  - 位置锁已解除
- 若条件不满足，CSP/IP 位置目标必须保持在当前反馈位置，速度/力矩目标不得越权透传。

### 3.6 故障后的行为

- `UpdateFromFeedback()` 在 `Armed/Running` 状态下检测到 fault 或 heartbeat 丢失时，必须降级到 `Faulted`。
- fault 恢复后不得自动清除“重新使能前重同步”要求。
- 故障恢复后的标准操作顺序固定为：
  - `recover -> enable -> resume`

## 4. 当前认可的验证点

以下回归不过，视为破坏安全契约：

- `OperationalCoordinator.RecoverThenEnableReleaseSucceedsAfterFeedbackHealthy`
- `OperationalCoordinator.ReleaseRejectedWhenAxisUnhealthy`
- `OperationalCoordinator.EnableRejectedWhenGlobalFaultLatched`
- `CanopenRobotHwRos.CommandSyncSequenceForcesGuardResetAndPositionResync`
- `CiA402Protocol.HaltToRunAdvancesEpochAndRelocks`
- `CiA402Protocol.RunRequiresValidEpochBeforeUnlock`
- `CiA402SM.FirstOperationEnabledStaysLockedUntilValidEpochCommand`
- `CiA402SM.EpochMismatchKeepsLocked`

建议回归命令：

```bash
ctest -R "^OperationalCoordinator\\.|^CiA402Protocol\\.|^AxisLogicTest\\.|^RobotHw\\.|^SharedState\\.|^CanopenRobotHwRos\\." --output-on-failure
```

## 5. 非目标

本文档不要求：

- `docs/archive/2026-04-19_deprecated/0324import.md` 中所有状态终态原文逐字落地；
- 立即删除所有旧兼容接口；
- 一次性完成全部架构收敛。

本文档只冻结一件事：当前系统必须继续保持已经验证过的稳定性和安全性，尤其是“使能前同步位置”与“健康门控后才允许运行”。
