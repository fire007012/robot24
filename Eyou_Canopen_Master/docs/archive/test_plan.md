# CANopen 主站测试方案（面向 8/10 与 3 小时稳定运行）

日期：2026-03-18

---

## 1. 目标与通过标准

目标：
- 功能正确：主流程、配置生效、状态机、PDO 校验可验证。
- 稳定运行：在较好环境连续运行 3 小时无崩溃、无死锁、无不可恢复故障。
- 故障自恢复：Heartbeat 丢失、EMCY、节点重启场景有明确恢复行为。
- 工程质量：形成 CI 自动回归门禁。

总体通过标准（Go/No-Go）：
- 单元/集成/HIL/Soak 测试全部通过。
- 3 小时 Soak 期间：
  - 0 崩溃、0 卡死、0 不可恢复故障；
  - 故障注入后可在约定时限内恢复到可运行态；
  - 无明显内存泄漏趋势；
  - 关键日志与指标可追溯。

## 2. 测试分层

### 2.1 单元测试（本地 + CI）

覆盖模块：
- `SharedState`
- `CiA402StateMachine`
- `joints_config`
- 单位换算
- PDO diff 逻辑

目的：
- 快速发现逻辑回归，提供分钟级反馈。

### 2.2 集成测试（无硬件，Mock 驱动）

覆盖链路：
- 启动-运行-退出全流程
- 配置读取到运行行为的端到端链路
- 故障注入后的状态迁移与恢复策略

目的：
- 在不依赖硬件情况下验证关键行为闭环。

### 2.3 HIL 测试（有硬件台架）

覆盖对象：
- `can0` 实际总线行为
- PDO/SDO/Heartbeat/EMCY
- 节点上下电、重启、短时中断

目的：
- 验证真实设备与总线下的系统表现。

### 2.4 Soak 长稳测试（3 小时）

覆盖重点：
- 长时运行稳定性
- 资源占用趋势
- 周期抖动与恢复耗时

目的：
- 作为“可上线前”最关键的稳定性验收项。

## 3. 用例矩阵（最小必测）

### 3.1 启动与配置

1. DCF 存在/不存在  
- 预期：存在时正常启动；不存在时明确报错并退出。

2. joints.yaml 合法/缺字段/非法 node_id  
- 预期：合法启动；非法配置给出明确错误（避免静默降级）。

3. 参数一致性检查（轴数、node_id、DCF 对齐）  
- 预期：不一致时阻断启动并给可定位提示。

### 3.2 配置生效性

1. `counts_per_rev/rated_torque_nm/velocity_scale/torque_scale`  
- 预期：修改配置后，读写换算结果按预期变化。

2. 状态机与故障策略参数  
- 预期：修改阈值/重试次数/保持周期后，状态机行为发生对应变化。

### 3.3 状态机行为

1. 启用链路  
- NotReady -> SwitchOnDisabled -> ReadyToSwitchOn -> OperationEnabled

2. 故障链路  
- FaultReactionActive/Fault 进入、复位边沿、重试上限、永久故障分支

### 3.4 PDO 映射校验

1. `verify_pdo_mapping=true` 且映射一致  
- 预期：轴可进入 operational。

2. `verify_pdo_mapping=true` 且映射不一致  
- 预期：阻断该轴 operational，并输出差异。

3. 校验超时  
- 预期：进入失败态并给明确日志，不允许“假成功”。

### 3.5 故障恢复

1. Heartbeat 超时与恢复  
- 预期：超时后降级；恢复后按策略回到可运行。

2. EMCY 注入  
- 预期：按错误类型执行可恢复/不可恢复策略并可观测。

3. 单节点重启  
- 预期：系统行为可预测，恢复路径明确。

### 3.6 安全退出

1. Ctrl+C 或终止信号  
- 预期：按设计顺序执行关机控制字流程并正常退出。

## 4. 3 小时 Soak 方案

测试时长：
- 连续 180 分钟。

运行条件：
- 主循环 100Hz。
- 记录系统与业务指标。
- 每 30 分钟注入 1 次可恢复故障（建议：短时 heartbeat 中断或单节点重启）。

采集指标（至少）：
- 进程存活状态
- CPU 与 RSS 内存
- 线程数
- 控制周期统计（平均/P95/P99/最大）
- 错误计数（EMCY、heartbeat timeout）
- 恢复次数与恢复耗时

通过标准：
- 0 崩溃、0 死锁、0 不可恢复故障
- 所有注入故障均在约定时限内恢复
- 资源曲线稳定，无明显泄漏或抖动失控

## 5. CI 回归方案

### 5.1 每次提交必跑

1. 构建：
```bash
cmake -S . -B build
cmake --build build -j
```

2. 测试：
```bash
ctest --test-dir build --output-on-failure
```

3. 静态检查（至少一种）：
- `clang-tidy` 或 `cppcheck`

门禁规则：
- 任一项失败即阻断合并。

### 5.2 每日定时任务

- 跑一轮扩展集成测试（含故障注入）。
- 产出日报：失败率、耗时、资源趋势、最近回归点。

---

## 6. 测试执行顺序建议

1. 先修通基础质量底座（CTest、CI、并发安全）。
2. 再补故障闭环（Heartbeat/EMCY/恢复策略）。
3. 再做配置生效与一致性校验。
4. 最后执行 3 小时 Soak 并出验收报告。
