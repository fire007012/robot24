# Eyou_ROS1_Master 统一入口使用说明（2026-04-09）

## 1. 正式入口

`Eyou_ROS1_Master` 在 facade 模式下的正式入口是：

- launch：`Eyou_ROS1_Master/launch/hybrid_motor_hw.launch`
- node：`hybrid_motor_hw_node`

推荐直接通过 launch 启动，不再分别手动拉起：

- `can_driver_node`
- `canopen_hw_ros_node`

当使用统一外观层时，生命周期 authority 应保持在 `hybrid_motor_hw_node` 一侧。

同时要明确边界：

- `can_driver` 的电机/硬件参数由 `can_driver` 包维护
- `Eyou_Canopen_Master` 的主站/关节参数由 `Eyou_Canopen_Master` 包维护
- `Eyou_ROS1_Master` 只负责组合、统一 service、统一 `controller_manager` 与 launch 入口

也就是说，`Eyou_ROS1_Master` 不应成为底层电机参数的第二份维护位置。

---

## 2. 最小启动示例

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  controllers_file:=$(rospack find Eyou_Canopen_Master)/config/ros/controllers.yaml \
  start_controllers:='joint_state_controller arm_position_controller' \
  stopped_controllers:=arm_velocity_controller \
  auto_init:=true \
  auto_enable:=true \
  auto_release:=true
```

说明：

- `controllers_file` 负责加载 controller 参数
- `start_controllers` 会由 `controller_manager/spawner` 直接拉起
- `stopped_controllers` 会以 `--stopped` 方式加载但不进入 running

如果要让 `hybrid` 直接暴露统一的 IP executor action 入口，推荐使用：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  use_ip_executor:=true \
  ip_executor_action_ns:=/arm_position_controller/follow_joint_trajectory \
  start_controllers:=joint_state_controller \
  auto_init:=true \
  auto_enable:=true \
  auto_release:=true
```

说明：

- `use_ip_executor=true` 时，`FollowJointTrajectory` 由 `hybrid` 内部 executor 接管
- 正式 action namespace 为 `/arm_position_controller/follow_joint_trajectory`
- 这一路径应由 `hybrid_motor_hw_node` 统一对外暴露，而不是重新回到 backend 原生节点

如果只想保留统一节点、不自动拉 controller，可设置：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  spawn_controllers:=false \
  auto_init:=false
```

## 2.1 完整启动流程

以下流程以当前三电机联调目标为准：

- `shoulder_yaw_joint`：`Eyou_Canopen_Master`
- `shoulder_pitch_joint`：`Eyou_Canopen_Master`
- `elbow_pitch_joint`：`can_driver`

推荐流程如下。

### 步骤 0：准备 CAN 口

若现场尚未拉起 CAN 口，可先执行：

```bash
sudo ip link set can0 down 2>/dev/null || true
sudo ip link set can1 down 2>/dev/null || true
sudo ip link set can0 type can bitrate 1000000
sudo ip link set can1 type can bitrate 1000000
sudo ip link set can0 up
sudo ip link set can1 up
```

检查：

```bash
ip -details link show can0
ip -details link show can1
```

### 步骤 1：source 工作空间

```bash
source /home/dianhua/Robot24_catkin_ws/devel/setup.bash
```

### 步骤 2：启动 unified 节点

推荐当前联调用法：

```bash
roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch \
  use_ip_executor:=true \
  ip_executor_action_ns:=/arm_position_controller/follow_joint_trajectory \
  start_controllers:=joint_state_controller \
  auto_init:=false \
  auto_enable:=false \
  auto_release:=false
```

说明：

- 当前推荐把 lifecycle 手动控制，便于先做模式/零点/限位等调试
- `use_ip_executor=true` 时，动作入口走 `hybrid` 内部 executor
- `start_controllers` 当前通常只需要 `joint_state_controller`

### 步骤 3：检查基础入口

检查 controller：

```bash
rosservice call /controller_manager/list_controllers
```

检查关键服务：

```bash
rosservice list | grep -E 'hybrid_motor_hw_node/(init|enable|disable|resume|shutdown|set_joint_mode|set_joint_zero|apply_joint_limits)'
```

检查 joint state：

```bash
rostopic echo -n 1 /joint_states
```

应至少看到：

- `shoulder_yaw_joint`
- `shoulder_pitch_joint`
- `elbow_pitch_joint`

### 步骤 4：执行 lifecycle bringup

```bash
rosservice call /hybrid_motor_hw_node/init "{}"
rosservice call /hybrid_motor_hw_node/enable "{}"
rosservice call /hybrid_motor_hw_node/resume "{}"
```

如果本轮需要先做模式切换或零点设置，建议在 `init` 后先不要 `resume`，
而是先执行：

```bash
rosservice call /hybrid_motor_hw_node/disable "{}"
```

然后进入 `Standby/disable` 语义下的模式/零点操作。

### 步骤 5：模式切换（可选）

当前统一模式入口：

```bash
rosservice call /hybrid_motor_hw_node/set_joint_mode "{joint_name: 'shoulder_yaw_joint', mode: 'position'}"
rosservice call /hybrid_motor_hw_node/set_joint_mode "{joint_name: 'shoulder_yaw_joint', mode: 'velocity'}"
rosservice call /hybrid_motor_hw_node/set_joint_mode "{joint_name: 'elbow_pitch_joint', mode: 'position'}"
rosservice call /hybrid_motor_hw_node/set_joint_mode "{joint_name: 'elbow_pitch_joint', mode: 'velocity'}"
```

注意：

- 统一模式名当前只提供 `position` 与 `velocity`
- 实际 backend 模式号由 `joint_mode_mappings.yaml` 决定
- 模式切换前建议先 `disable`

### 步骤 6：零点 + 限位（可选，推荐分两步）

统一当前点设零：

```bash
rosservice call /hybrid_motor_hw_node/set_joint_zero "{joint_name: 'shoulder_yaw_joint', zero_offset_rad: 0.0, use_current_position_as_zero: true}"
rosservice call /hybrid_motor_hw_node/set_joint_zero "{joint_name: 'shoulder_pitch_joint', zero_offset_rad: 0.0, use_current_position_as_zero: true}"
rosservice call /hybrid_motor_hw_node/set_joint_zero "{joint_name: 'elbow_pitch_joint', zero_offset_rad: 0.0, use_current_position_as_zero: true}"
```

统一应用 URDF 限位：

```bash
rosservice call /hybrid_motor_hw_node/apply_joint_limits "{joint_name: 'shoulder_yaw_joint', min_position_rad: 0.0, max_position_rad: 0.0, use_urdf_limits: true, require_current_inside_limits: false}"
rosservice call /hybrid_motor_hw_node/apply_joint_limits "{joint_name: 'shoulder_pitch_joint', min_position_rad: 0.0, max_position_rad: 0.0, use_urdf_limits: true, require_current_inside_limits: false}"
rosservice call /hybrid_motor_hw_node/apply_joint_limits "{joint_name: 'elbow_pitch_joint', min_position_rad: 0.0, max_position_rad: 0.0, use_urdf_limits: true, require_current_inside_limits: false}"
```

注意：

- 两侧零点/限位写入都建议在 `disable` 后执行
- 当前统一语义是“zero-only + apply-limits-only”
- 首装调机推荐先 `set_joint_zero`，确认零点正确后再 `apply_joint_limits`

### 步骤 7：进入运行态

完成模式/零点操作后：

```bash
rosservice call /hybrid_motor_hw_node/enable "{}"
rosservice call /hybrid_motor_hw_node/resume "{}"
```

### 步骤 8：启动 UI

```bash
rosrun Eyou_ROS1_Master hybrid_joint_action_ui.py
```

只做配置核对时：

```bash
rosrun Eyou_ROS1_Master hybrid_joint_action_ui.py --dry-run
```

### 步骤 9：动作联调

当前统一动作入口：

- action namespace：`/arm_position_controller/follow_joint_trajectory`

可直接用 UI，也可以用 action client / MoveIt。

### 步骤 10：结束联调

```bash
rosservice call /hybrid_motor_hw_node/halt "{}"
rosservice call /hybrid_motor_hw_node/shutdown "{}"
```

## 3. Launch 参数

### 3.1 后端配置

- `canopen_dcf_path`：CANopen 主站 DCF 路径
- `canopen_joints_path`：CANopen `joints.yaml` 路径
- `can_driver_config`：`can_driver` 关节配置
- `joint_mode_mappings_file`：统一模式路由映射 YAML
- `loop_hz`：主循环频率

默认来源：

- `canopen_dcf_path` 默认指向 `Eyou_Canopen_Master/config/master.dcf`
- `canopen_joints_path` 默认指向 `Eyou_Canopen_Master/config/joints.yaml`
- `can_driver_config` 默认指向 `can_driver/config/can_driver.yaml`
- `joint_mode_mappings_file` 默认指向 `Eyou_ROS1_Master/config/joint_mode_mappings.yaml`

其中：

- `Eyou_ROS1_Master` 会把 `can_driver` 原生配置加载到自己的私有命名空间下
- 但 `CanDriverHW` 的初始化是通过 `can_driver_node` 子私有命名空间句柄接入的
- 因此 `master` 复用的是 `can_driver` 原生参数契约，而不是另维护一份平铺副本
- `can_driver` 用哪个 CAN 口，来自各 joint 的 `can_device`
- `canopen` 用哪个 CAN 口，来自 `joints.yaml` 中的 `canopen.interface`
- 若需要 loopback 测试，`can_driver` 侧从自己的 `init_loopback` 配置读取，而不是由 `master` 单独定义全局参数

### 3.2 自动启动参数

- `auto_init`
- `auto_enable`
- `auto_release`
- `use_ip_executor`
- `ip_executor_action_ns`
- `ip_executor_rate_hz`

约束：

1. `auto_enable=true` 或 `auto_release=true` 时，必须同时 `auto_init=true`
2. `auto_release=true` 时，必须同时 `auto_enable=true`

语义：

1. `auto_init=false`：不自动推进生命周期
2. `auto_init=true`：统一经由 hybrid facade 执行 `init`，成功后两侧进入 `Armed`
3. `auto_enable=true`：接受“目标至少到 Armed”的声明；在当前实现里，`init` 成功后本来就已是 `Armed`
4. `auto_release=true`：在前述基础上继续推进到 `Running`
5. `use_ip_executor=true`：启用 `hybrid` 内部 `IpFollowJointTrajectoryExecutor`
6. `ip_executor_action_ns`：正式 action 路径，推荐使用绝对路径
7. `ip_executor_rate_hz`：executor 指令更新频率，默认继承 `loop_hz`

### 3.3 Controller Spawner 参数

- `controllers_file`：controller 参数 YAML；为空则不加载
- `spawn_controllers`：是否启动 spawner，默认 `true`
- `start_controllers`：启动后直接进入 running 的 controller 列表，默认 `joint_state_controller`
- `stopped_controllers`：仅加载不启动的 controller 列表

补充说明：

- 默认 `controllers_file` 由 `Eyou_ROS1_Master` 提供最小配置，只保证 `joint_state_controller` 可加载
- 这属于组合层默认入口配置，不属于底层电机参数配置
- 若需要位置/速度控制器，应显式传入业务侧 controller YAML
- 若 `use_ip_executor=true`，通常只需要 `joint_state_controller` 进入 running

## 4. 对外服务

统一服务都挂在 `hybrid_motor_hw_node` 私有命名空间下：

- `~init`
- `~enable`
- `~disable`
- `~halt`
- `~resume`
- `~recover`
- `~shutdown`
- `~set_joint_mode`
- `~set_zero`
- `~set_zero_limit`

其中：

- `~init`：完成统一 init，并在首次成功后执行 post-init hook
- `~resume`：统一从 `Armed -> Running`
- 重复 `~init`：返回 `already initialized`，不会再次执行 post-init hook
- `~set_joint_mode`：统一模式设置入口，使用 `joint_name + mode` 语义
- `~set_zero`：CANopen 关节当前点设零；若启用自动软限位，会同时重写软限位
- `~set_zero_limit`：can_driver 关节当前点设零并按 URDF/请求写限位

### 4.1 统一模式服务

建议通过：

- `~set_joint_mode`

而不是直接使用 backend 原生模式号。

当前默认提供两种统一模式：

- `position`
- `velocity`

它们在 backend 的实际路由值由：

- `Eyou_ROS1_Master/config/joint_mode_mappings.yaml`

决定。当前默认映射为：

```yaml
joint_mode_mappings:
  modes:
    position:
      canopen: 8
      can_driver: 0
    velocity:
      canopen: 9
      can_driver: 1
```

这意味着：

- `canopen` 的 `position` 默认路由到 `CSP(8)`
- `canopen` 的 `velocity` 默认路由到 `CSV(9)`
- `can_driver` 的 `position` 默认路由到 `CMD_SET_MODE value=0`
- `can_driver` 的 `velocity` 默认路由到 `CMD_SET_MODE value=1`

如果后续需要调整实际模式号，应优先修改 YAML，而不是改 UI 或 `hybrid` 代码。

## 5. Facade 模式下不建议直接使用的入口

当系统已经通过 `hybrid_motor_hw.launch` 进入 facade 模式后，不建议再直接操作：

- `can_driver` 原生 lifecycle service
- `Eyou_Canopen_Master` 原生 lifecycle service
- CANopen 原生 auto startup 序列

原因是这些入口会绕开 hybrid authority，破坏“两边一起成功或一起失败”的收敛语义。

---

## 6. 失败时的预期收敛

当前统一入口的故障语义如下：

1. hybrid lifecycle 某一步后半段失败且 rollback 再失败时，会进入 fail-safe shutdown，尽量回到 `Configured`
2. post-init hook 失败时，会执行统一 `shutdown` 回滚
3. 非法 auto 参数组合会在节点启动阶段直接失败，不进入半启动状态
4. 重复 `init` 不会重复执行 hook
5. `shutdown` 采用 best-effort 语义：两侧 backend 都会被尝试执行 shutdown，再聚合错误返回

这意味着：

- “成功”要求两侧后端都达成目标态
- “失败”优先保证系统回到统一安全态，而不是保留半成功状态
- 即使其中一侧 shutdown 失败，另一侧也不应被跳过

---

## 7. 建议使用方式

推荐顺序：

1. 通过 `hybrid_motor_hw.launch` 启动统一节点
2. 用 launch 参数决定是否 auto startup、是否自动拉 controller
3. 后续所有生命周期操作只走 `hybrid_motor_hw_node` 的统一服务

如果要做联调或故障排查，优先看：

- `hybrid_motor_hw_node` 日志
- `controller_manager/list_controllers`
- `~/init/~resume/~shutdown` 的返回消息

如果需要复用现有 UI，建议使用：

```bash
rosrun Eyou_ROS1_Master hybrid_joint_action_ui.py
```

排查配置时可先执行：

```bash
rosrun Eyou_ROS1_Master hybrid_joint_action_ui.py --dry-run
```

默认行为：

- 自动合并 `Eyou_Canopen_Master/config/joints.yaml`
- 自动合并 `can_driver/config/can_driver.yaml`
- 默认 `service_ns` 为 `/hybrid_motor_hw_node`
- 默认 `action_ns` 为 `/arm_position_controller/follow_joint_trajectory`
- 模式设置走 `~set_joint_mode`
- 模式列表默认来自 `joint_mode_mappings.yaml`
- 提供两个独立按钮：
  - “当前点设零” -> `~/set_joint_zero`
  - “应用URDF限位” -> `~/apply_joint_limits`

注意：

- 两侧模式切换、零点/限位写入都要求目标轴处于 `Standby/disable` 语义，建议先执行 `disable`

## 8. 必要 API

以下 API 是当前三电机 unified 联调的必要入口。

### 8.1 Lifecycle

```bash
rosservice call /hybrid_motor_hw_node/init "{}"
rosservice call /hybrid_motor_hw_node/enable "{}"
rosservice call /hybrid_motor_hw_node/disable "{}"
rosservice call /hybrid_motor_hw_node/halt "{}"
rosservice call /hybrid_motor_hw_node/resume "{}"
rosservice call /hybrid_motor_hw_node/recover "{}"
rosservice call /hybrid_motor_hw_node/shutdown "{}"
```

### 8.2 统一模式设置

服务类型：

- `Eyou_ROS1_Master/SetJointMode`

请求：

- `joint_name`
- `mode`

响应：

- `success`
- `message`
- `backend`
- `mapped_mode`

示例：

```bash
rosservice call /hybrid_motor_hw_node/set_joint_mode "{joint_name: 'shoulder_yaw_joint', mode: 'position'}"
rosservice call /hybrid_motor_hw_node/set_joint_mode "{joint_name: 'elbow_pitch_joint', mode: 'velocity'}"
```

### 8.3 CANopen 零点设置

服务类型：

- `Eyou_Canopen_Master/SetZero`

请求：

- `axis_index`

响应：

- `success`
- `message`

示例：

```bash
rosservice call /hybrid_motor_hw_node/set_zero "{axis_index: 0}"
```

### 8.4 can_driver 零点 + 限位设置

服务类型：

- `can_driver/SetZeroLimit`

关键请求字段：

- `motor_id`
- `use_current_position_as_zero`
- `use_urdf_limits`
- `apply_to_motor`

示例：

```bash
rosservice call /hybrid_motor_hw_node/set_zero_limit "motor_id: 19
zero_offset_rad: 0.0
use_current_position_as_zero: true
min_position_rad: 0.0
max_position_rad: 0.0
use_urdf_limits: true
apply_to_motor: true"
```

### 8.5 动作入口

action namespace：

- `/arm_position_controller/follow_joint_trajectory`

使用者：

- `hybrid_joint_action_ui.py`
- `actionlib` client
- MoveIt

### 8.6 UI

启动：

```bash
rosrun Eyou_ROS1_Master hybrid_joint_action_ui.py
```

配置核对：

```bash
rosrun Eyou_ROS1_Master hybrid_joint_action_ui.py --dry-run
```
