# Eyou_ROS1_Master 统一外观层实施计划

## 背景与目标

当前系统存在两套独立的电机控制包：

- **`can_driver`**：雷赛 MT / 意优 PP 协议，实现 `CanDriverHW`（继承 `hardware_interface::RobotHW`）
- **`Eyou_Canopen_Master`**：CANopen / Lely / CiA402 协议，实现 `CanopenRobotHwRos`（继承 `hardware_interface::RobotHW`）

两套后端各自有独立的节点入口、`controller_manager`、服务接口和生命周期状态机，上层难以统一控制。

**本包目标**：在单个节点内同时托管两套后端，共享一个 `controller_manager`，对外暴露统一的服务接口和 `/joint_states`，上层完全不感知底层差异。

> 参考上游设计文档：[统一外观层实施方案](../../docs/统一外观层实施方案.md)

---

## 设计原则

- 不统一底层协议实现，两套后端各走各的通信链路
- 不统一原生 YAML，各包保留自己的配置文件格式
- 两个旧节点（`can_driver_node`、`canopen_hw_ros_node`）**保留不动**，本包是额外新增
- `HybridRobotHW` 不拥有两个子 `RobotHW`，由 `main` 持有并传指针进去
- joint 名全局唯一，`init()` 时校验，重复则启动失败
- 任一后端初始化失败，节点整体退出，不做部分降级运行

---

## 架构

```
Eyou_ROS1_Master/
├── hybrid_motor_hw_node              ← 节点入口 (main)
├── HybridRobotHW                     ← 组合 RobotHW，聚合两个子 RobotHW
│   ├── CanDriverHW                   ← 来自 can_driver（直接复用）
│   └── CanopenRobotHwRos             ← 来自 Eyou_Canopen_Master（直接复用）
├── HybridOperationalCoordinator      ← 统一状态机，协调两边 OperationalCoordinator
└── HybridServiceGateway              ← 统一 ROS 服务入口
```

依赖方向（单向，两个被依赖包不感知本包存在）：

```
Eyou_ROS1_Master → can_driver
Eyou_ROS1_Master → Eyou_Canopen_Master
```

## 组件设计

### HybridRobotHW

继承 `hardware_interface::RobotHW`，`init()` 时：

1. 调用 `CanDriverHW::init()`
2. 调用 `CanopenRobotHwRos` 对应的初始化
3. 合并两边的 `hardware_interface` 注册到同一个 `controller_manager`
4. 校验所有 joint 名全局唯一

`read()` / `write()` 串行调用两边。

相关源码：
- [`can_driver/include/can_driver/CanDriverHW.h`](../../can_driver/include/can_driver/CanDriverHW.h)
- [`Eyou_Canopen_Master/include/canopen_hw/canopen_robot_hw_ros.hpp`](../../Eyou_Canopen_Master/include/canopen_hw/canopen_robot_hw_ros.hpp)

### HybridOperationalCoordinator

持有两个协调器指针：
- `can_driver::OperationalCoordinator*`
- `canopen_hw::OperationalCoordinator*`

每个请求串行调用两边，任一失败则整体失败。对外暴露的 `mode()` 取两边状态的悲观值（最低状态）。

状态集与两边保持一致：
`Inactive → Configured → Standby → Armed → Running`
`Running ←→ Armed`（Halt/Resume）
`任意 → Faulted → Recovering → Standby`

相关源码：
- [`can_driver/include/can_driver/operational_coordinator.hpp`](../../can_driver/include/can_driver/operational_coordinator.hpp)
- [`Eyou_Canopen_Master/include/canopen_hw/operational_coordinator.hpp`](../../Eyou_Canopen_Master/include/canopen_hw/operational_coordinator.hpp)

### HybridServiceGateway

注册统一 ROS 服务（全部使用 `std_srvs/Trigger`，init 参数通过 rosparam 传入）：

| 服务名 | 语义 |
|---|---|
| `~init` | 建链并完成启动检查，成功后进入 Armed |
| `~enable` | Standby → Armed |
| `~resume` | Armed → Running |
| `~halt` | Running → Armed |
| `~disable` | 回到 Standby |
| `~recover` | Faulted → Standby |
| `~shutdown` | 停止通信 |

相关源码：
- [`can_driver/include/can_driver/lifecycle_service_gateway.hpp`](../../can_driver/include/can_driver/lifecycle_service_gateway.hpp)
- [`Eyou_Canopen_Master/include/canopen_hw/service_gateway.hpp`](../../Eyou_Canopen_Master/include/canopen_hw/service_gateway.hpp)

### 主循环

```cpp
// CANopen 侧需要额外的 feedback/intent 步骤
canopen_coord.UpdateFromFeedback();
canopen_coord.ComputeIntents();

hybrid_hw.read(now, dt);
cm.update(now, dt);
hybrid_hw.write(now, dt);
```

## 验证方式

1. `catkin_make` 编译通过，无符号未解析、无头文件缺失
2. `roslaunch Eyou_ROS1_Master hybrid_motor_hw.launch`（无硬件时用 loopback/vcan）启动无崩溃
3. `rosservice call /hybrid_motor_hw_node/init` 返回成功
4. `rostopic echo /joint_states` 能看到来自两边的关节状态
5. 状态机转换：`enable` → `resume` → `halt` → `disable` → `recover` 均正常

## 非目标

- 不统一原生 YAML 结构
- 不重写底层协议栈
- 不替换现有两个独立节点
- 不做部分降级运行（任一后端失败则整体失败）
