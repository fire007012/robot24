# 三电机联调修复实施方案（2026-04-09）

## 1. 目的

本文档用于把“三电机实机联调前问题报告与修复方案”的后续实施动作，
收敛为可直接执行的 commit 级修复清单。

目标不是再讨论“问题是否存在”，而是明确：

1. 每个 commit 要改什么
2. 每个 commit 为什么单独存在
3. 每个 commit 的测试与验收口径是什么

---

## 2. 当前结论

截至当前代码状态：

1. `Eyou_ROS1_Master` 的统一 bringup / lifecycle / `/joint_states` 基础链路已具备
2. 三电机统一动作联调链路尚未收口
3. 需要补齐的关键缺口仍然是：
   - `can_driver` 总线切到 `can1`
   - `hybrid` executor 动态纳入 `can_driver` 关节
   - `hybrid` 的正式 action/UI 入口收口

因此修复顺序必须遵循：

1. 先修物理总线配置
2. 再补 `can_driver` 的 executor 参数来源
3. 再补 `hybrid` 侧统一 executor 配置
4. 最后收口 launch 和 UI 入口

---

## 3. Commit 1

### 提交信息

`fix(can_driver): move elbow joint to can1`

### 目标

保证当前三电机实机联调的物理总线拓扑正确：

- `Eyou_Canopen_Master -> can0`
- `can_driver -> can1`

### 修改范围

- `can_driver/config/can_driver.yaml`

### 具体修改

仅修改：

- `elbow_pitch_joint.can_device: can0 -> can1`

本 commit 不夹带 parser、executor 或 launch 逻辑修改。

### 原因

这一项直接影响 `hybrid ~/init` 最终命中的物理 CAN 口。
如果不先修，后续动作联调和抓包结论都不可信。

### 验收

1. `git diff` 只出现 `can_driver.yaml` 中该处配置变化
2. `hybrid` 通过 `can_driver` joints 参数解析得到的设备名应为 `can1`
3. 现场抓包时，`elbow_pitch_joint` 初始化流量出现在 `can1`

## 4. Commit 2

### 提交信息

`feat(can_driver): add per-joint IP executor limits`

### 目标

让 `can_driver` 侧与 `Eyou_Canopen_Master` 一样，
具备每轴独立的 Ruckig / IP executor 约束来源。

### 修改范围

- `can_driver/include/can_driver/CanDriverHwTypes.h`
- `can_driver/include/can_driver/JointConfigParser.h`
- `can_driver/src/JointConfigParser.cpp`
- `can_driver/src/CanDriverHW.cpp`
- `can_driver/config/can_driver.yaml`
- `can_driver/tests/test_joint_config_parser.cpp`

### 具体修改

给 `can_driver` 关节配置增加 4 个可选字段：

- `ip_max_velocity`
- `ip_max_acceleration`
- `ip_max_jerk`
- `ip_goal_tolerance`

数据链路要求：

1. `ParsedJointConfig` 增加这 4 个字段
2. `CanDriverJointConfig` 增加这 4 个字段
3. `JointConfigParser` 负责解析和基本合法性校验
4. `CanDriverHW::parseAndSetupJoints()` 将解析结果写入运行时 joint 配置
5. `can_driver.yaml` 为 `elbow_pitch_joint` 增加这 4 个字段

默认值与 `Eyou_Canopen_Master` 保持一致：

- `ip_max_velocity = 1.0`
- `ip_max_acceleration = 2.0`
- `ip_max_jerk = 10.0`
- `ip_goal_tolerance = 0.001`

### 原因

如果没有这一步，`hybrid` 即使后续能读取 `can_driver` 关节列表，
也拿不到统一 executor 所需的每轴运动约束。

### 测试

扩展：

- `can_driver/tests/test_joint_config_parser.cpp`

至少覆盖：

1. 未配置时回退默认值
2. 显式配置时能正确解析
3. 非法值时报错

### 验收

1. `test_joint_config_parser` 通过
2. `can_driver` 编译通过
3. `elbow_pitch_joint` 在配置层和运行时层都能拿到 4 个 IP 约束字段

## 5. Commit 3

### 提交信息

`feat(Eyou_ROS1_Master): build hybrid IP executor config from both backends`

### 目标

让 `hybrid` 的 `IpFollowJointTrajectoryExecutor`
真正覆盖当前全部轨迹关节，而不是只覆盖 CANopen 两轴。

### 修改范围

- `Eyou_ROS1_Master/include/Eyou_ROS1_Master/hybrid_ip_executor_config.hpp`（新增）
- `Eyou_ROS1_Master/src/hybrid_ip_executor_config.cpp`（新增）
- `Eyou_ROS1_Master/src/hybrid_motor_hw_node.cpp`
- `Eyou_ROS1_Master/tests/test_hybrid_ip_executor_config.cpp`（新增）
- `Eyou_ROS1_Master/CMakeLists.txt`

同时更新外层 `src` 仓库引用的 `can_driver` 子模块版本到 Commit 2。

### 具体修改

将当前直接写在 `hybrid_motor_hw_node.cpp` 里的 executor 组装逻辑抽成独立 helper。

helper 应动态合并两侧配置：

1. 从 `master_cfg.joints` 读取 CANopen 侧 joint 与约束
2. 从 `~can_driver_node/joints` 参数读取 `can_driver` 侧 joint 与约束
3. 只纳入具备位置轨迹语义的关节
4. 检查重复 joint name
5. 生成统一的 `IpFollowJointTrajectoryExecutor::Config`

设计约束：

1. 不在 `Eyou_ROS1_Master` 里硬编码“三电机列表”
2. 不直接解析磁盘上的 `can_driver.yaml`
3. 复用已加载到 `hybrid_motor_hw_node` 私有命名空间下的原生参数契约

### 原因

当前 executor 的 `exec_cfg` 只从 `master_cfg.joints` 构造，
本质上仍是“两轴 executor + hybrid RobotHW 指针”，
还不是真正的三轴 unified executor。

### 测试

新增：

- `Eyou_ROS1_Master/tests/test_hybrid_ip_executor_config.cpp`

至少覆盖：

1. 2 个 CANopen 关节 + 1 个 can_driver 关节成功合并
2. 跨后端 duplicate joint name 报错
3. 非位置语义 joint 不进入 executor 配置

### 验收

1. `use_ip_executor=true` 时 `exec_cfg.joint_names` 包含：
   - `shoulder_yaw_joint`
   - `shoulder_pitch_joint`
   - `elbow_pitch_joint`
2. 每个 joint 都有完整约束向量
3. 三轴 goal 不再因 joint count mismatch 被拒绝

## 6. Commit 4

### 提交信息

`fix(Eyou_ROS1_Master): expose hybrid IP executor action path in launch`

### 目标

把 `hybrid` 在 `use_ip_executor=true` 场景下的正式动作入口整理清楚，
避免仍沿用后端原生入口或混淆 JTC / executor 语义。

### 修改范围

- `Eyou_ROS1_Master/launch/hybrid_motor_hw.launch`
- `Eyou_ROS1_Master/config/controllers_minimal.yaml`
- `Eyou_ROS1_Master/tests/test_hybrid_launch_ip_executor.test`（新增）
- `Eyou_ROS1_Master/tests/test_hybrid_launch_ip_executor.cpp`（新增）
- `Eyou_ROS1_Master/CMakeLists.txt`
- `Eyou_ROS1_Master/docs/22_统一入口使用说明_2026-04-09.md`

### 具体修改

在 launch 中正式暴露：

- `use_ip_executor`
- `ip_executor_action_ns`
- `ip_executor_rate_hz`

并透传给 `hybrid_motor_hw_node`。

文档层需要明确区分：

1. `use_ip_executor=false`
   - `arm_position_controller` 由 controller_manager/JTC 负责
2. `use_ip_executor=true`
   - `FollowJointTrajectory` 由 `IpFollowJointTrajectoryExecutor` 接管
   - 默认只需保证 `joint_state_controller` 运行

### 原因

当前 `hybrid` 默认配置只保证 `joint_state_controller`；
而联调用法文档里仍容易让人把 `arm_position_controller` 和 executor 混在一起理解。

### 测试

新增 launch smoke：

- 启动 `hybrid_motor_hw.launch use_ip_executor:=true`
- 确认：
  - `joint_state_controller` 为 `running`
  - action server 对应 topic/service 已出现

### 验收

1. `use_ip_executor:=true` 时可清晰确认正式动作入口
2. 不需要 backend 原生节点即可拿到 `hybrid` action path
3. 文档示例与实际 launch 语义一致

1. 定位：
   - `Eyou_Canopen_Master/config/joints.yaml`
   - `can_driver/config/can_driver.yaml`
2. 生成临时 merged joints yaml
3. 默认调用原始 `joint_action_ui.py`
4. 默认参数改为：
   - `--service-ns /hybrid_motor_hw_node`
   - `--action-ns /arm_position_controller/follow_joint_trajectory`

- `joints[].name`
- 展示用途的 `node_id`

建议同时提供：

- `--dry-run`

用于打印最终解析出的路径、joint 集、service/action namespace 后退出，
便于现场排查。

### 原因

现有 UI 默认仍面向 `/canopen_hw_node`，
直接使用会把联调入口重新绕回 backend 原生节点。

### 验收

   能看到 3 个 joint
2. 默认 service namespace 为 `/hybrid_motor_hw_node`
3. 默认 action namespace 为 `/arm_position_controller/follow_joint_trajectory`
4. UI 中零点设置应按 joint backend 自动路由到：
   - `set_zero`
   - `set_zero_limit`

## 8. 后续补充：统一模式服务

在完成前 5 个 commit 后，还需要补一层 facade 级统一模式路由，
避免 `hybrid` UI 继续沿用 backend 原生模式号。

### 建议提交信息

`feat(Eyou_ROS1_Master): add unified joint mode routing`

### 目标

1. UI 不再直接暴露 `7/8/9/10`
2. `can_driver` 不再被错误套用 `canopen` 模式号
3. 后端模式映射改为通过 YAML 管理

### 修改范围

- 新增 `Eyou_ROS1_Master/srv/SetJointMode.srv`
- 新增 `Eyou_ROS1_Master/config/joint_mode_mappings.yaml`
- 新增 `HybridModeRouter`
- 更新 `hybrid_joint_action_ui.py`

### 统一语义

当前先只提供：

- `position`
- `velocity`

默认 YAML：

```yaml
joint_mode_mappings:
  modes:
    position:
      canopen: 8
      can_driver: 0
    velocity:
      canopen: 9
      can_driver: 1
```

这样后续若需要改 backend 实际模式值，只改 YAML 即可。
